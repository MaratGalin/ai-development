# Engineering Guardrails: CI/CD, Testing, Linting, Automation

## Почему guardrails стали критичнее в эпоху AI

До активного внедрения AI-команда могла жить с частично ручным контролем качества: часть ошибок ловилась на ревью, часть, уже в эксплуатации, часть, не ловилась вовсе. С приходом AI это перестало быть безопасной стратегией.

Причина простая, AI ускоряет не только полезную работу, но и распространение слабых практик. Если в проекте уже есть неоднозначные паттерны, пропущенные проверки и «временные» исключения из правил, AI будет воспроизводить их с высокой скоростью. Этот эффект можно назвать **amplification effect**: растет скорость доставки, но одновременно растет скорость масштабирования ошибок.

Поэтому guardrails, инженерные ограничители и автоматические проверки, становятся не «дополнением к зрелому процессу», а базовым условием безопасного AI-ускорения. Иначе команда получает короткий всплеск velocity, а затем платит за него регрессиями, архитектурным дрейфом и дорогим rework.

Практический вывод такой: чем больше доля AI в ежедневной разработке, тем выше должна быть доля автоматизированного контроля на каждом PR и на каждом merge.

## Минимальный набор guardrails

Ниже, минимальный набор, который стоит считать обязательным независимо от стека и размера команды. Конкретные инструменты выбираются под текущий toolchain компании, важно не название инструмента, а роль проверки.

### 1) Linting

Цель linting, сделать код предсказуемым и заранее ловить типовые ошибки и anti-patterns.

Что должно быть обеспечено:
- единый набор правил для репозитория или для домена
- одинаковый результат локально и в CI
- блокировка merge при критичных нарушениях

### 2) Formatting

Форматирование снимает бессмысленные споры на ревью и уменьшает шум в diff.

Что важно:
- автоформат при коммите или перед push
- единые правила во всех сервисах, где это возможно
- отсутствие ручных исключений без явной причины

### 3) Type checking

Статическая проверка типов критична в AI-потоке, потому что AI часто генерирует правдоподобный, но слабо согласованный код на границах модулей.

Что дает type checking:
- раннее обнаружение несовместимых контрактов
- снижение числа runtime-ошибок после merge
- более надежные рефакторинги

### 4) Тесты

Минимум, unit + integration-тесты для критичных сценариев. Не обязательно сразу стремиться к экстремальному coverage, но критические user flows должны быть защищены.

Ключевые принципы:
- тесты обязаны быть стабильными, flaky-тесты подрывают доверие ко всей системе
- AI-изменение без тестового доказательства считается рискованным по умолчанию
- приоритет, проверка бизнес-инвариантов, а не только «счастливого пути»

### 5) CI

CI должен быть единой точкой принятия решения о качестве изменения.

Минимальная последовательность в pipeline:
1. сборка
2. linting + formatting check
3. type check
4. тесты

Если любой шаг падает, merge блокируется. Исключения допускаются редко и фиксируются явно, чтобы не превращаться в «тихую норму».

## Автоматическое принуждение к качеству

Guardrails работают только тогда, когда они обязательны по процессу, а не остаются «рекомендацией». Для этого нужен слой автоматического enforcement.

### Pre-commit hooks

На уровне разработчика hooks позволяют отсеять очевидные нарушения до открытия PR:
- запуск быстрых линтеров
- автоформатирование
- простые структурные проверки

Это снижает шум в CI и экономит время ревьюеров.

### CI checks

В CI должна выполняться полная, воспроизводимая проверка. В отличие от локального окружения, это нейтральная и обязательная среда контроля. Именно она защищает от «у меня локально прошло».

### Structural tests

Часть правил невозможно надежно выразить обычными линтерами. Для архитектурных инвариантов нужны структурные тесты или custom linters. Они проверяют не стиль, а форму системы.

Примеры архитектурных инвариантов:
- слой UI не обращается напрямую к persistence-слою
- доменный слой не зависит от инфраструктурного
- запрещены импорты между bounded context, если нет разрешенного контракта

Такие проверки особенно ценны при AI-генерации, потому что они ловят архитектурный дрейф в момент появления, а не через несколько спринтов.

## Code review для AI-generated code, что добавить к стандартному процессу

Обычное ревью остается обязательным, но для AI-сгенерированных изменений нужен дополнительный чеклист. Цель, не «наказать» разработчика, а закрыть типовые риски машинной генерации.

Минимальные дополнительные требования:

1. **Запрет на `as any` без формального исключения**  
   Любое ослабление типобезопасности должно иметь обоснование и план устранения.

2. **Запрет на пустой `catch`**  
   Ошибки не должны проглатываться. Нужны либо корректная обработка, либо явный проброс, либо логирование по принятому стандарту.

3. **Запрет `console.log` в production-коде**  
   Диагностика должна идти через принятый механизм логирования с контролем уровней и формата.

4. **Проверка соответствия архитектурным ограничениям**  
   Даже «рабочий» код отклоняется, если нарушает границы модулей, слоев и контрактов.

5. **Проверка покрывающих тестов для измененного поведения**  
   Для каждого существенного изменения должен быть тест, который подтверждает новое поведение или защищает от регрессии.

6. **Проверка прозрачности намерения**  
   В PR должно быть ясно, что меняется, почему это делается и какие риски приняты.

Важно, эти требования не привязаны к одному конкретному инструменту. Они совместимы с существующей инфраструктурой команды и могут внедряться поэтапно.

## Как внедрять guardrails без конфликта с существующим toolchain

Сильная ошибка при внедрении guardrails, попытка сразу заменить все инструменты команды «идеальным» стеком. На практике это дает сопротивление и снижает скорость.

Рабочий путь:
- зафиксировать обязательные роли проверок, а не бренды инструментов
- картировать текущий toolchain команды на эти роли
- закрыть пробелы минимальными изменениями
- постепенно усиливать правила после стабилизации базового набора

Так команда сохраняет операционную устойчивость и одновременно повышает качество AI-изменений.

## Decision: Which guardrails to implement first

Если внедрять все сразу невозможно, приоритет должен быть таким:

1. **CI как обязательный gate для merge** с минимальным пайплайном: build, lint/format check, type check, tests.
2. **Pre-commit hooks** для быстрых локальных проверок и автоформата.
3. **Дополнительный AI-review checklist** с явными запретами: `as any` без исключения, пустой `catch`, `console.log` в production.
4. **Structural tests/custom linters** для 2-3 главных архитектурных инвариантов.

Итоговое решение для CTO и engineering leadership: сначала обеспечить обязательный автоматический контроль качества на пути к merge, затем закрепить AI-специфичные правила ревью, после этого масштабировать структурные архитектурные проверки. Такой порядок дает максимальное снижение риска при минимальном организационном трении.
