# Executive Summary

## Problem
Большинство компаний уже запустили инициативы в GenAI для software engineering, но не получают системного эффекта на уровне delivery. Главная причина не в модели, а в операционной модели внедрения: пилоты изолированы, ownership размыт, quality gates не адаптированы под AI-generated code. В результате руководители видят активность, но не видят масштабируемого результата в скорости и качестве разработки.

## Opportunity
Рынок показывает, что AI agents уже работают в production при правильной организации процесса. Потенциал для CTO и VP Engineering лежит в переходе от точечных экспериментов к управляемой модели, где AI встроен в SDLC: от написания кода и тестов до review и сопровождения. Практический фокус, единые метрики и staged rollout снижают риск «дорогого пилота без продолжения».

## Key Numbers
- OpenAI Harness: команда из 3 engineers сгенерировала 1M LOC, при этом manual coding составил 0, средняя скорость достигла 3.5 PR на инженера в день, reported speedup около 10x (source: OpenAI Harness case data).
- Stripe Minions: более 1000 merged PR в неделю от AI agents (source: Stripe engineering case data).
- Anthropic: около 90% кода Claude Code написано самим Claude Code (source: Anthropic engineering statement).
- 95% AI pilot initiatives не доходят до масштабирования (source: MIT, GenAI Divide).
- 56% CEOs сообщают, что AI efforts дали «nothing» в бизнес-результате (source: PwC, CEO Survey 2026).

## Action Plan
Рекомендуемый путь внедрения для engineering-организации: **Audit → Pilot (4 weeks) → Scale (3 months)**.

1. **Audit**: зафиксировать baseline по cycle time, PR throughput, defect leakage, review latency, а также определить зоны с высоким повторением задач.
2. **Pilot (4 weeks)**: выбрать 1-2 продуктовые команды, ограничить scope, ввести policy для AI usage, обязательные quality checks и weekly executive review.
3. **Scale (3 months)**: расширять только подтвержденные практики, стандартизировать guardrails, обновить роль Tech Leads и Staff Engineers как владельцев adoption.

## Expected Impact
При дисциплинированном внедрении можно ожидать рост инженерной пропускной способности и сокращение time-to-merge, но эффект зависит от governance, качества данных и зрелости команды. Корректно ожидать диапазон улучшений по операционным метрикам, а не фиксированный ROI. Цель на горизонте 3 месяцев, получить повторяемый процесс, где AI повышает скорость поставки без компромисса по качеству и безопасности.

## Key Decisions to Make
- Какие engineering metrics станут официальными KPI для AI adoption на уровне VP Engineering.
- Какие типы задач разрешены для AI-first execution, а какие требуют human-only ownership.
- Какой набор quality and security gates обязателен перед merge AI-generated изменений.
- Кто персонально владеет программой внедрения, budget и межфункциональной синхронизацией.
- По каким критериям принимается решение о переходе от Pilot к Scale.

# Industry Context: текущее состояние AI в software development

## (a) Leader cases

Ниже собраны кейсы компаний и практиков, которые уже встроили AI-агентов в ежедневную инженерную работу. Важный акцент: речь не про демо, а про повторяемый production-процесс с измеримыми показателями.

### 1) OpenAI Harness Engineering

**Ключевые цифры**
- Команда выросла с **3 до 7 инженеров**.
- За примерно 5 месяцев создан внутренний продукт объемом около **1M LOC**.
- Принципиально зафиксирован подход **0 строк ручного кода** для основной реализации.
- Средний поток разработки достиг **3.5 PR на инженера в день**.

Источники:
- OpenAI Harness Engineering: https://www.engineering.fyi/article/harness-engineering-leveraging-codex-in-an-agent-first-world
- Разбор с цитатами и метриками: https://www.ignorance.ai/p/the-emerging-harness-engineering

**Что они сделали**
- Перестроили роль инженера: меньше ручного кодинга, больше проектирования среды, в которой агент может стабильно работать.
- Ввели жесткие архитектурные ограничения и проверку этих ограничений линтерами и структурными тестами.
- Сформировали регулярный цикл улучшений: каждая типовая ошибка агента превращается в правило, тест, инструкцию или tooling.

**Уроки**
- Производительность растет не от модели сама по себе, а от качества инженерной среды вокруг модели.
- Архитектурная дисциплина и автоматические проверки становятся не бюрократией, а множителем скорости.
- Главный дефицит смещается из «умения писать код» в «умение строить систему управления агентами».

### 2) Stripe Minions

**Ключевые цифры**
- В Stripe внутренние агенты Minions дают **1000+ merged PR в неделю**.
- Основной поток запускается прямо из Slack, затем агент сам проходит путь до PR с CI и ревью.

Источник:
- Официальный пост Stripe: https://stripe.dev/blog/minions-stripes-one-shot-end-to-end-coding-agents

**Что они сделали**
- Подключили агентов к внутренним инструментам компании через централизованный слой (Toolshed/MCP-подход).
- Запустили агентов в изолированных devboxes, близких к среде инженеров.
- Сместили человека в точку архитектурного контроля и code review, а не в точку ручного исполнения каждой задачи.

**Уроки**
- Масштаб возможен, когда агент встроен в стандартный инженерный контур, а не живет в отдельной «песочнице экспериментов».
- Интеграция с корпоративными инструментами критична. Без нее AI остается ассистентом, а не частью delivery-конвейера.
- В крупных организациях качество процесса важнее качества единичного промпта.

### 3) Anthropic и практики long-running harness

**Ключевые цифры**
- По данным Addy Osmani, внутри Anthropic около **90% кода Claude Code пишется Claude Code**.
- В исследовательской практике Anthropic для сложных задач использовалась декомпозиция на **200+ фич** (пример в их инженерной статье).

Источники:
- Addy Osmani, workflow 2026: https://addyosmani.com/blog/ai-coding-workflow/
- Anthropic, Effective harnesses: https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents

**Что они сделали**
- Разделили процесс на инициализирующего агента и рабочих агентов.
- Ввели обязательные артефакты состояния: feature list, progress file, git-коммиты как «точки передачи смены».
- Принудили инкрементальную разработку: один шаг, проверка, фиксация состояния, следующий шаг.

**Уроки**
- Для многошаговых задач без «памяти процесса» агент деградирует по качеству и начинает терять контекст.
- Структурированная среда с явными артефактами делает сессии взаимозаменяемыми и снижает хаос.
- Для enterprise это прямой сигнал: без формализации handoff-процесса масштабирование AI-команд будет нестабильным.

### 4) OpenClaw, Peter Steinberger

**Ключевые цифры**
- Со слов Peter Steinberger, темп достигал **6600+ commits за месяц** в режиме solo-разработки.
- Одновременно в работе обычно **5-10 агентных сессий**.

Источники:
- Интервью Pragmatic Engineer: https://newsletter.pragmaticengineer.com/p/the-creator-of-clawd-i-ship-code
- Аналитический разбор с теми же цифрами: https://www.ignorance.ai/p/the-emerging-harness-engineering

**Что он сделал**
- Поставил управление параллельными агентами в центр процесса.
- Радикально поднял ценность этапа планирования до запуска генерации.
- Удержал архитектурный контроль на уровне продукта, даже когда не читает каждую строку кода вручную.

**Уроки**
- Параллельные агенты дают кратный выигрыш только при сильном архитектурном руководстве.
- Узкое место переносится с «написать код» на «проверить направление и качество».
- Такой режим требует зрелого инженерного вкуса и строгой ответственности за системные решения.

### 5) Addy Osmani (Google Chrome team): операционная модель AI-native workflow

**Ключевые акценты и цифры**
- В своей практике 2026 Addy фиксирует дисциплину: план и спецификация до кода, итеративные чанки, обязательная валидация и частые git-checkpoints.
- В тексте также подтверждается маркер рынка про **~90% кода Claude Code, написанного Claude Code** как сигнал сдвига роли инженера.

Источник:
- https://addyosmani.com/blog/ai-coding-workflow/

**Что сделано**
- Унифицирован процесс «spec first -> plan -> incremental execution -> verify».
- Явно встроены review, testing, linting и возврат к контролю инженера на каждом шаге.

**Уроки**
- «Быстро писать» без «быстро проверять» создает скрытый долг и уводит метрики в ложный рост.
- Стабильный эффект появляется там, где AI встроен в инженерную дисциплину, а не подменяет ее.

## (c) The Paradox: почему при массовом внедрении 95% пилотов не доходят до production

В 2026 году сформировался явный парадокс.

- Доступ к инструментам есть почти у всех.
- Расходы растут рекордными темпами.
- Локальные демо и пилоты выглядят убедительно.
- Но большая часть инициатив не масштабируется.

### Ключевые цифры парадокса

1. **95% GenAI-пилотов не доходят до production**.
   - Указано в Enterprise AI 2026 guide со ссылкой на MIT GenAI Divide.
   - Источник: https://ssntpl.com/enterprise-ai-implementation-complete-2026-guide/

2. **56% CEO сообщают, что не получили результата от AI-инициатив**.
   - Указано в Enterprise AI 2026 guide со ссылкой на PwC CEO Survey 2026.
   - Источники:
     - https://ssntpl.com/enterprise-ai-implementation-complete-2026-guide/
     - https://www.pwc.com/gx/en/issues/c-suite-insights/ceo-survey.html

3. На уровне инженерных команд может расти активность, но не расти итоговая ценность.
   - Типичный паттерн: больше PR и больше кода, но одновременно рост rework и снижение maintainability.
   - Источник: https://www.synlabs.io/post/ai-roi-software-engineering

### Корневые причины

1. **Ошибка масштаба**: пилот проектируется как эксперимент, а не как репетиция production.
2. **Ошибка метрик**: команды меряют output (PR, LOC, токены), а не outcome (time-to-value, defect leakage, rework).
3. **Ошибка governance**: ownership размыт между IT, продуктом и безопасностью.
4. **Ошибка процесса**: отсутствует контур «правило -> тест -> верификация -> обновление harness».
5. **Ошибка оргдизайна**: роли team lead/staff engineer не перестроены под управление агентами и quality gates.

### Практический смысл парадокса для нашей компании

Если мы повторим типовой путь «дать доступ к AI и ждать роста скорости», то почти наверняка попадем в группу неудачных пилотов. Чтобы войти в меньшинство успешных программ, нужен другой фокус:

- не инструмент first, а operating model first;
- не «демо-скорость», а управляемая производственная дисциплина;
- не разовые инициативы, а системная работа с quality и процессом.

## (d) Harness Engineering как рабочая концепция для enterprise

### Определение

**Harness Engineering** это проектирование управляемой среды, в которой AI-агенты способны стабильно выполнять инженерные задачи с предсказуемым качеством. Сама модель важна, но решающий фактор, это набор ограничений, инструментов, правил, тестов, документации и обратной связи.

Ключевые источники:
- OpenAI Harness Engineering: https://www.engineering.fyi/article/harness-engineering-leveraging-codex-in-an-agent-first-world
- Emerging playbook: https://www.ignorance.ai/p/the-emerging-harness-engineering
- Build the harness, not the code: https://vitthalmirji.com/2026/02/build-the-harness-not-the-code
- Anthropic long-running harnesses: https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents

### Из чего состоит harness на практике

1. **Архитектурные границы**
   - Явные слои и запреты на недопустимые зависимости.
   - Автоматическая проверка этих правил в CI.

2. **Tooling и доступ к среде**
   - Агенту доступны те же инструменты, что и инженеру, но в контролируемом контуре.
   - Стандартизированные команды сборки, тестов, линтинга, проверки безопасности.

3. **Документация как операционный артефакт**
   - AGENTS.md, runbooks, progress logs, feature lists.
   - Документация обновляется при каждом системном классе ошибки.

4. **Verification loop**
   - Каждый шаг агента завершается проверкой: unit/integration/e2e, статический анализ, policy checks.
   - Ошибка превращается в новое правило, чтобы не повторяться.

5. **Роли и ответственность**
   - Нужен владелец агентной операционной модели на уровне команды или стрима.
   - Code ownership и архитектурное решение остаются на человеке.

### Почему это важно для нас

Harness Engineering дает управленческий язык для перехода от пилотов к production. Мы начинаем обсуждать не «какая модель сильнее», а:

- какие ограничения обязательны,
- какие задачи разрешены для AI-first,
- какие проверки блокируют merge,
- как мы меряем реальный эффект,
- кто принимает решение о масштабировании.

Именно этот переход разделяет лидеров и компании, которые застревают в режиме экспериментов.

## Decision: Which approach for our company, evolutionary vs revolutionary

### Вариант 1: Revolutionary (резкий AI-first для всего SDLC)

**Плюсы**
- Максимальная скорость изменений в коротком горизонте.
- Быстрый сигнал рынку и руководству, что компания делает ставку на новый уклад.

**Риски**
- Высокая вероятность потери управляемости качества.
- Рост rework, если governance и harness не готовы.
- Конфликт с текущими процессами безопасности и compliance.

### Вариант 2: Evolutionary (пошаговое масштабирование)

**Плюсы**
- Контролируемый риск.
- Быстрое накопление внутренних практик и стандартов.
- Возможность измерять эффект по этапам и корректировать курс.

**Риски**
- Более медленная видимая динамика в первые месяцы.
- Требуется дисциплина в приоритезации, чтобы не распылиться.

### Рекомендация для нашей компании

С учетом отраслевой статистики и наблюдаемого парадокса 95% провалов, для нас рационален путь **Evolutionary with strong harness**:

1. Запустить ограниченный AI-first контур на 1-2 командах с высоким объемом повторяемых задач.
2. Зафиксировать обязательные quality gates до масштабирования.
3. Ввести единый набор метрик outcome: cycle time, defect leakage, review latency, rework ratio, доля задач с успешным agent-first исполнением.
4. Масштабировать только те практики, которые показали устойчивый эффект минимум 2 спринта подряд.

**Итоговое решение:**

> Для нашей компании рекомендован **эволюционный подход**. Не «внедрять AI везде сразу», а последовательно строить harness и расширять зону автономии агентов только после подтвержденного качества и экономического эффекта.

# Что такое AI-Ready Codebase

## Определение

**AI-Ready Codebase** это кодовая база, в которой и люди, и AI-ассистенты могут быстро понять контекст, безопасно внести изменение и проверить результат без догадок.

Обычная «хорошая» кодовая база может быть удобной только для команды, которая давно в проекте. AI-Ready состояние строже: структура, правила, документация и quality gates должны быть понятны новому инженеру и машине в первый день.

Коротко:

**AI-Ready = Human-Friendly + Machine-Navigable + Well-Documented Intent**

Где:
- **Human-Friendly**: разработчик быстро находит нужный код и понимает, как принято работать в проекте.
- **Machine-Navigable**: агент может пройти по репозиторию, выбрать правильный путь и не размножать случайные паттерны.
- **Well-Documented Intent**: в репозитории явно зафиксированы архитектурные решения, поведение фич и продуктовые цели.

## 7 ключевых характеристик AI-Ready codebase

Ниже минимальный checklist, который должен быть выполнен системно, а не точечно:

- [ ] **One obvious way** для типовых задач. Для каждой частой операции есть один предпочтительный путь реализации.
- [ ] **Тесты защищают критические user flows**. Важные сценарии пользователя ломаются первыми в тестах, а не в production.
- [ ] **Build, lint, test запускаются без догадок**. Команды явно задокументированы и повторяемы в локальной среде и CI.
- [ ] **Deprecated patterns явно помечены**. Старые подходы не выглядят как «нормальный пример для копирования».
- [ ] **Дубли и конфликтующие абстракции сокращены**. В репозитории нет нескольких почти одинаковых «правильных» реализаций одной задачи.
- [ ] **Документация описывает архитектуру, поведение фич и продуктовые цели**. Есть связь между кодом, решением и бизнес-смыслом.
- [ ] **CI ловит типовые регрессии до merge**. Основные проверки качества автоматизированы и обязательны.

## AI Readiness Maturity Model (5 уровней)

Модель нужна, чтобы не спорить «мы готовы или нет», а объективно определить текущий уровень и следующий шаг.

### Level 1, Chaotic

**Сигналы**:
- Нет единого стиля реализации, много случайных паттернов.
- Критические flows слабо покрыты тестами или тесты нестабильны.
- Команды сборки и проверки отличаются между командами.

**Риск при использовании AI**: агент ускоряет накопление техдолга, потому что копирует хаос.

### Level 2, Basic

**Сигналы**:
- Есть базовые стандарты кода и частичная документация.
- Локально можно запустить build/lint/test, но процесс не всегда воспроизводим.
- CI есть, но покрывает только часть рисков.

**Риск при использовании AI**: локальные улучшения есть, но масштабирование ломается между командами.

### Level 3, Structured

**Сигналы**:
- Зафиксированы основные паттерны, deprecated-зоны отмечены.
- Критические flows покрыты тестами на приемлемом уровне.
- Документация по архитектуре и ключевым фичам поддерживается в актуальном состоянии.
- CI стабильно блокирует типовые ошибки до merge.

**Эффект при использовании AI**: заметный рост скорости разработки без резкого роста дефектов.

### Level 4, Optimized

**Сигналы**:
- Паттерны унифицированы по всей инженерной организации.
- Контекст для AI задан явно, например AGENTS.md, инструкции по запуску и проверке, правила по deprecated-практикам.
- Метрики по quality и delivery используются для регулярного улучшения процесса.

**Эффект при использовании AI**: AI-ассистенты работают предсказуемо, доля rework снижается.

### Level 5, AI-Native

**Сигналы**:
- Процессы разработки изначально проектируются с учетом human + AI collaboration.
- Архитектурные и продуктовые решения быстро транслируются в код, тесты и документацию.
- Guardrails, CI и engineering policy обновляются как часть стандартного цикла изменений.

**Эффект при использовании AI**: высокая скорость поставки и управляемое качество на уровне всей организации.

Ответьте «Да» или «Нет» по каждому пункту.

- **0-2 «Да»**: вы на уровне **Chaotic**.
- **3-4 «Да»**: вы на уровне **Basic**.
- **5 «Да»**: вы на уровне **Structured**.
- **6 «Да»**: вы на уровне **Optimized**.
- **7 «Да»**: вы на уровне **AI-Native**.

## Почему это системный подход, а не «просто пишите тесты»

Тесты критичны, но сами по себе не решают проблему навигации, согласованности паттернов и явного контекста для AI. По материалам Dakic, LLMx, Hypercube и Propel, устойчивый эффект появляется только при комбинации четырех факторов:

1. Ясная структура и предсказуемые паттерны.
2. Проверяемость изменений через тесты и CI.
3. Явные инструкции для разработки и сопровождения.
4. Актуальная документация намерений, а не только API и синтаксиса.

Именно эта связка снижает риск, когда AI ускоряет не качество, а хаос.

## Decision: What level is your company and what's the next step

Определите текущий уровень зрелости по чеклисту выше и зафиксируйте один следующий шаг на ближайшие 2-4 недели.

- Если вы на **Chaotic/Basic**, следующий шаг: стандартизировать команды build/lint/test и закрыть тестами 2-3 критических user flow.
- Если вы на **Structured**, следующий шаг: формализовать AI-контекст (AGENTS.md, правила deprecated-паттернов, критерии готовности PR).
- Если вы на **Optimized**, следующий шаг: масштабировать практики между командами через единые engineering policy и метрики.
- Если вы на **AI-Native**, следующий шаг: регулярно пересматривать guardrails и архитектурные соглашения, чтобы скорость не разрушала качество.

Решение для CTO простое: сначала поднять базовую инженерную предсказуемость, потом ускорять delivery с AI на всей организации.

# Architecture: Vertical Slices and AI-Friendly Patterns

Архитектура влияет на эффективность AI-средств сильнее, чем обычно ожидают команды. Модель может быть мощной, промпт может быть хорошим, но если код разложен так, что для одного изменения нужно открыть двадцать файлов в разных слоях, качество ответов быстро падает. Причина простая, у агента ограничено окно контекста, а навигация по репозиторию стоит времени и токенов.

В классической оценке архитектуры мы смотрим на масштабируемость, сопровождаемость, производительность и надежность. В 2025 году добавился еще один критерий, насколько легко AI понимает систему и безопасно вносит изменения. Этот критерий не заменяет остальные, но уже напрямую влияет на скорость поставки.

## Почему архитектура определяет результат AI

AI-ассистент не «видит» проект целиком. Он читает его частями, строит локальную карту и постоянно рискует потерять связь между деталями. Поэтому для него критичны три вещи:

1. **Локальность контекста.** Чем больше кода фичи лежит рядом, тем выше шанс, что модель поймет задачу без догадок.
2. **Предсказуемая навигация.** Когда структура повторяется от фичи к фиче, агент быстрее находит нужные места.
3. **Малое число файлов на одно изменение.** Если изменение расползается по многим директориям, растут ошибки интеграции и регрессии.

Ключевой вопрос для архитектурной оценки под AI:

> **«How many files must AI open to change feature X?»**

Если ответ «много и в разных слоях», то даже хороший агент начнет давать шум. Если ответ «несколько файлов внутри одной фичи», результат обычно стабилен.

## Сравнение архитектур через призму AI

Ниже сравнение четырех подходов, которые чаще всего встречаются в продуктовой разработке.

| Архитектура | Оценка AI-friendly | Что помогает AI | Что мешает AI |
|---|---|---|---|
| **Vertical Slice** | **★★★★★** | Код сгруппирован по бизнес-фичам, высокая локальность, удобно загружать один срез целиком | Риск дублирования и расползания общих утилит между срезами |
| **Layered** | **★★☆☆☆** | Четкое разделение технических ролей, понятна базовая схема | Для одной фичи надо идти через UI, service, domain, repository, mapping, много прыжков и потерь контекста |
| **Microservices** | **★★★☆☆** | Границы сервисов уменьшают размер локального контекста | Сложность зависит от внутренней структуры сервиса, межсервисные контракты и распределенные изменения могут перегружать AI |
| **Pipes & Filters** | **★★★★☆** | Линейный поток обработки, понятные этапы, хорошо для pipeline-задач | Плохо подходит для сложной продуктовой логики с богатым UI и пересекающимися сценариями |

### Короткий вывод по таблице

- **Vertical Slice** дает лучший баланс для feature-разработки с AI.
- **Layered** остается рабочим вариантом для небольших систем, но чаще всего дорог по контексту.
- **Microservices** не «хороши сами по себе», все решает внутренняя организация каждого сервиса.
- **Pipes & Filters** очень удобен там, где бизнес-задача естественно выглядит как последовательность этапов обработки данных.

## Почему Vertical Slice обычно выигрывает

Подход Vertical Slice группирует код по изменяемости. Это главное. Мы меняем фичи, а не «слой репозиториев» или «слой сервисов». Когда весь путь запроса лежит рядом, AI проще построить причинно-следственную цепочку от API или UI до правил и хранения.

Практический эффект:

- меньше файлов в контексте на один запрос,
- меньше ложных связей между независимыми частями,
- меньше необходимости держать в голове абстракции «на весь проект».

Идея из практики Vertical Slice хорошо сочетается с CQRS-подходом, где команда рассматривает каждый use case отдельно. Это снижает давление «общих сервисов на все случаи жизни» и оставляет больше пространства для точечных решений внутри среза.

## Пример структуры Vertical Slice

Ниже пример структуры, где каждая фича содержит свой полный путь, API, правила, доступ к данным, тесты и локальную документацию.

```text
src/
  features/
    billing/
      create-invoice/
        endpoint.ts
        handler.ts
        validator.ts
        domain/
          invoice.ts
          pricing-policy.ts
        infra/
          invoice-repository.ts
          sql.ts
        dto/
          request.ts
          response.ts
        tests/
          create-invoice.unit.test.ts
          create-invoice.integration.test.ts
        README.md
      send-reminder/
        endpoint.ts
        handler.ts
        scheduler.ts
        infra/
          mail-client.ts
        tests/
          send-reminder.integration.test.ts
    profile/
      update-avatar/
        endpoint.ts
        handler.ts
        image-policy.ts
        infra/
          object-storage.ts
        tests/
          update-avatar.unit.test.ts
  shared/
    auth/
    observability/
    idempotency/
```

Что здесь критично для AI:

1. Папка среза названа по use case, а не по технологии.
2. Все файлы сценария лежат рядом.
3. Тесты находятся около кода фичи.
4. Есть явная граница `shared`, куда попадают только действительно общие механизмы.

## Как архитектура влияет на context window на практике

Рассмотрим задачу, «добавить обязательный флаг подтверждения при создании счета и вернуть новое поле в ответе».

В Layered-структуре это часто затрагивает:

- controller,
- dto,
- service,
- domain model,
- repository,
- mapper,
- тесты в отдельной иерархии.

То есть агенту приходится собирать контекст из большого числа разнесенных мест. Ошибка в одном переходе дает каскад исправлений.

В Vertical Slice-структуре обычно меняются 4-7 файлов внутри одного каталога use case плюс тесты рядом. Агент реже теряет нить, а reviewer быстрее проверяет корректность.

Именно поэтому вопрос про число файлов на изменение полезен как KPI архитектуры под AI.

## Caveats, когда Vertical Slice применять не стоит или стоит частично

Vertical Slice не является универсальным лекарством. Есть сценарии, где полное применение может навредить.

### 1) Shared kernels с жесткими инвариантами

Если в системе есть общий доменный «ядро-модуль» с критичными правилами, например pricing core для всей платформы, бездумное дублирование логики по срезам опасно. В таком случае ядро должно оставаться единым, а срезы работают через четко определенный API ядра.

### 2) Performance-critical участки

В горячих путях, где важны микросекунды и контроль аллокаций, архитектура может подчиняться профилированию, а не удобству навигации. Здесь допустимы специализированные модули с ручной оптимизацией, если границы и причины задокументированы.

### 3) Legacy core с высоким риском

Если система большая и старая, миграция «одним махом» может сломать поставку. Для таких случаев лучше инкрементальный подход, новые фичи сразу в Vertical Slice, старые части переводятся по мере изменения.

### 4) Сильная платформенная стандартизация

Иногда платформа требует строго единого каркаса, например централизованный workflow, регуляторные ограничения, общий security gateway. Тогда часть логики остается платформенной, а срезы строятся внутри разрешенных рамок.

## Рекомендации по внедрению без радикализма

Чтобы получить выигрыш для AI и не потерять управляемость, лучше идти по шагам.

1. **Оценить текущие фичи по метрике файлов на изменение.** Выбрать 2-3 самых болезненных сценария.
2. **Сделать пилотный срез.** Перенести одну фичу в Vertical Slice, не трогая весь проект.
3. **Зафиксировать правила.** Именование, структура папок, границы `shared`, требования к тестам.
4. **Добавить локальную документацию.** Короткий README внутри среза, где описаны цель и ограничения.
5. **Измерить эффект.** Время на типовые изменения, число правок после ревью, число регрессий.

Такой путь убирает риск «архитектурной революции» и дает практические данные для следующего шага.

## Decision: Do we need to migrate architecture and how

**Да, миграция нужна, но не тотальная и не мгновенная.**

Предлагаемое решение:

- Принять **Vertical Slice** как целевую модель для новых продуктовых фич.
- Сохранить **исключения** для shared kernel, performance-critical модулей и стабильно работающих legacy-core зон.
- Применять **incremental migration**, переносить старые части только при реальном изменении функциональности.
- Ввести явный критерий готовности, для типовой задачи AI должен менять фичу, открывая минимальное число файлов в пределах одного среза.

Итоговый принцип простой, архитектура должна ускорять поставку и снижать риск ошибок AI, а не создавать догматический запрет на разумные исключения.

# Documentation: AGENTS.md, Context and Dual Audience Design

## Зачем этот раздел

Если команда хочет стабильно работать с AI-ассистентами в коде, ей мало просто открыть чат и дать задачу. Нужна рабочая документация, которая помогает модели быстро понять проект, не терять ограничения и не ломать архитектуру. В 2025 году это оформилось в отдельный класс артефактов: `CLAUDE.md`, `AGENTS.md`, вспомогательные контекстные файлы и ADR.

Главная идея простая: модель почти ничего не "помнит" между сессиями, если это явно не передано в контекст. Поэтому качество результата напрямую зависит от качества входного контекста.

Ниже, практичный подход, который можно внедрить сразу без лишней бюрократии.

## Что такое AGENTS.md и CLAUDE.md

`CLAUDE.md` появился как формат для Claude Code, `AGENTS.md` стал более универсальным вариантом для разных инструментов и сред. По сути это один и тот же тип документа: краткая оперативная карта проекта для AI-агента.

Хороший файл такого типа отвечает на три вопроса:

1. **HOW**: как работать в репозитории.
2. **WHY**: зачем проект существует и почему решения приняты именно так.
3. **WHAT**: что находится в репозитории, где что лежит и за что отвечает.

По материалу HumanLayer важный нюанс такой: эти файлы часто подмешиваются почти в каждую сессию, но модель может игнорировать часть инструкций, если считает их нерелевантными задаче. Это значит, что длинный и шумный документ ухудшает результат. Нужен короткий файл с универсальными правилами и ссылками на более узкие инструкции.

### Принцип "меньше, но точнее"

Распространенная ошибка, превратить AGENTS.md в "свалку" всех возможных команд, стандартов и исключений. Тогда в контексте появляется шум, модель хуже выполняет инструкции, и команда получает нестабильный результат.

Практичное правило:

- в корневом AGENTS.md только то, что почти всегда применимо,
- подробности выносить в отдельные документы,
- в AGENTS.md оставлять ссылки на эти документы и коротко объяснять, когда их читать.

Такой подход называют progressive disclosure: нужное знание подгружается по ситуации, а не льется в модель заранее целиком.

## Заполненный пример HOW/WHY/WHAT

Ниже, небольшой пример, как выглядит полезный, но не перегруженный файл.

```md
# AGENTS.md

## HOW
- Менеджер пакетов: pnpm (`pnpm install`, `pnpm -r test`, `pnpm -r build`).
- Перед PR обязательно: тесты, линт, typecheck.
- Не менять публичные API пакетов без обновления `docs/api-contracts.md`.
- Не добавлять новые зависимости без краткого обоснования в PR.

## WHY
- Продукт: платформа аналитики для B2B-команд продаж.
- Приоритет: надежность отчетов и трассируемость источников данных.
- Архитектурный выбор: модульный монолит, чтобы сохранять скорость поставки и простой деплой.

## WHAT
- `apps/web`: фронтенд (Next.js).
- `apps/api`: backend API и фоновые задачи.
- `packages/domain`: бизнес-логика и доменные модели.
- `packages/db`: схема и миграции.
- `docs/adr`: архитектурные решения и их причины.

## Read if relevant
- `docs/dev/testing.md` - стратегия тестов и команды для локального прогона.
- `docs/dev/release.md` - процесс релиза и чеклист.
- `docs/adr` - ограничения и принятые архитектурные решения.
```

Этот формат легко читать человеку и легко парсить модели. Он короткий, но содержит опорные точки, без которых агент почти всегда ошибается.

## Context Engineering и "context packing"

### Что это такое

Context engineering, это инженерная дисциплина подготовки входного контекста для модели перед работой. "Context packing" в этом контексте означает упаковку релевантных артефактов в компактный, структурированный набор: цели задачи, ограничения, нужные файлы, команды проверки, примеры, ссылки на решения.

Проще говоря, мы не просим модель "додумать" проект, мы даем ей рабочий набор фактов.

### Почему это важно

Если у модели мало контекста, она начинает заполнять пробелы вероятностными догадками. В коде это приводит к типовым сбоям:

- меняются не те модули,
- нарушаются локальные соглашения,
- игнорируются архитектурные ограничения,
- правки выглядят "разумно", но ломают совместимость.

Если контекст собран хорошо, модель чаще действует как аккуратный инженер: следует текущим паттернам, правильно выбирает слой изменений, не изобретает лишнюю структуру.

### Какие инструменты помогают

Из практик, описанных в инженерных workflow 2025 года, для упаковки контекста часто используют:

- **gitingest**: берет репозиторий и готовит текстовый контекст для LLM, чтобы быстро передать структуру и код.
- **repo2txt**: формирует единый текстовый слепок репозитория или выбранных частей.

Оба инструмента полезны, когда нужно быстро дать модели обзор большого проекта. Но есть ограничение: "дамп всего" не равен "хороший контекст". В рабочем потоке лучше делать гибрид:

1. Сначала обзорная упаковка (карта репозитория).
2. Затем точечное добавление только релевантных файлов под конкретную задачу.
3. Явное указание запретов и критериев готовности.

Иначе можно переполнить окно контекста и потерять сигнал в шуме.

## ADR: почему надо фиксировать "почему", а не только "что"

Архитектурные документы часто пишут как факт: "перешли на X". Для AI и для новых участников команды такой формат слабый, потому что не раскрывает мотивацию и ограничения.

**Architecture Decision Records (ADR)** решают это. Короткий ADR обычно включает:

- контекст проблемы,
- рассматриваемые варианты,
- выбранное решение,
- последствия и компромиссы.

### Практическая ценность ADR

1. **Снижение повторных споров.** Команда видит, почему выбор уже сделан.
2. **Защита от случайных откатов.** Агент не предлагает "вернуться назад", если понимает причину решения.
3. **Быстрый онбординг.** Новые разработчики и модели быстрее понимают ограничения.
4. **Качество изменений.** В PR проще проверить, не противоречит ли правка текущим решениям.

Для AI это особенно важно: без ADR модель может предлагать технически красивый, но контекстно неверный рефакторинг. Наличие ADR в "read if relevant" списке резко снижает такие ошибки.

## Dual Audience Design: документация сразу для людей и машин

Раньше документация адресовалась только человеку. Теперь ее читают два класса потребителей:

1. Люди, которым нужна ясность и объяснимость.
2. Модели, которым нужны четкие, однозначные, структурированные инструкции.

Dual audience design, это подход, где один документ работает для обеих сторон.

### Принципы dual-audience документа

- **Ясные заголовки и стабильная структура.**
- **Короткие буллеты для правил и команд.**
- **Минимум двусмысленных формулировок.**
- **Явные секции "Do" и "Do not".**
- **Ссылки на первоисточник вместо копий кода.**

### Что это дает в практике

- Человеку проще сканировать документ и находить ответы.
- Модель точнее извлекает инструкции и меньше "додумывает".
- Команда получает единый источник правил, без дублирования между wiki, чатами и устной памятью.

## Практичный шаблон AGENTS.md (готов к копированию)

Ниже шаблон, который можно брать как стартовую точку. Он нарочно простой.

```md
# AGENTS.md

## Purpose
Кратко: что делает продукт и для кого.

## HOW
### Workflow
- Установка: `<команда>`
- Запуск локально: `<команда>`
- Тесты: `<команда>`
- Линт / typecheck: `<команда>`
- Сборка: `<команда>`

### Conventions
- Изменения должны быть минимальными и локальными.
- Следовать существующим паттернам в кодовой базе.
- Новые зависимости добавлять только при необходимости и с обоснованием.

### Do not
- Не менять публичные контракты без явного согласования.
- Не делать массовый рефакторинг в задаче, где нужен точечный фикс.
- Не обходить проверки качества.

## WHY
### Product context
- Бизнес-цель 1
- Бизнес-цель 2

### Architecture constraints
- Ограничение 1 (почему важно)
- Ограничение 2 (почему важно)

### Key decisions
- Ссылка на ADR-001
- Ссылка на ADR-00N

## WHAT
### Repository map
- `apps/...` : назначение
- `packages/...` : назначение
- `docs/...` : назначение

### Critical modules
- Модуль A: за что отвечает, где точка входа
- Модуль B: за что отвечает, где точка входа

## Task intake checklist
- Цель задачи понятна и измерима.
- Известны файлы, которые можно менять.
- Известны критерии готовности.
- Указаны команды проверки.

## Read if relevant
- `docs/dev/testing.md`
- `docs/dev/deployment.md`
- `docs/adr/`
- `docs/domain-glossary.md`
```

### Как поддерживать шаблон в живом состоянии

- Пересматривать AGENTS.md раз в 2-4 недели.
- После каждого значимого архитектурного решения добавлять ADR и ссылку.
- Удалять устаревшие команды сразу, иначе агент будет запускать невалидные шаги.
- Держать файл коротким. Если секция распухает, выносить детали в отдельный документ.

## Минимальный набор документации для AI-ready команды

Чтобы не утонуть в документации, полезно зафиксировать "разумный минимум":

1. **AGENTS.md** в корне репозитория.
2. **Папка ADR** хотя бы с ключевыми решениями.
3. **Один документ по проверкам**: тесты, линт, сборка.
4. **Карта репозитория** на 1 страницу.
5. **Глоссарий доменных терминов** для снижения двусмысленности.

Этого достаточно, чтобы агент стабильно выдавал полезный код, а команда не тратила недели на поддержание тяжелого процесса.

## Типовые ошибки внедрения

- Пытаться описать в AGENTS.md вообще все процессы компании.
- Писать правила без проверяемых команд.
- Копировать большие куски кода в документацию вместо ссылок на исходник.
- Не обновлять документы после реальных изменений в проекте.
- Игнорировать ADR и хранить причины решений только в головах отдельных людей.

Если убрать эти ошибки в начале, эффективность AI-ассистентов растет заметно уже в первые недели.

## Decision: What documentation level to implement and minimum required

**Решение для внедрения в проекте:**

- Внедряем **средний, практичный уровень документации**: не минимализм "почти ничего", но и не тяжелая процессная система.
- Базовый контракт для всех задач с AI: актуальный `AGENTS.md` + проверяемые команды + ссылки на ADR.
- Для архитектурно чувствительных областей обязательно фиксируем решения в ADR с причиной и последствиями.
- Для больших задач применяем context packing: обзор репозитория + релевантные файлы + критерии готовности.

**Minimum required (обязательный минимум):**

1. `AGENTS.md` по структуре HOW/WHY/WHAT.
2. Не менее 3 ключевых ADR по текущей архитектуре.
3. Документ с командами проверки (test/lint/typecheck/build).
4. Короткая карта репозитория с критичными модулями.

Этот минимум закрывает потребности людей и машин одновременно, снижает число ложных правок, и дает устойчивую базу для следующих задач отчета.

# Engineering Guardrails: CI/CD, Testing, Linting, Automation

## Почему guardrails стали критичнее в эпоху AI

До активного внедрения AI-команда могла жить с частично ручным контролем качества: часть ошибок ловилась на ревью, часть, уже в эксплуатации, часть, не ловилась вовсе. С приходом AI это перестало быть безопасной стратегией.

Причина простая, AI ускоряет не только полезную работу, но и распространение слабых практик. Если в проекте уже есть неоднозначные паттерны, пропущенные проверки и «временные» исключения из правил, AI будет воспроизводить их с высокой скоростью. Этот эффект можно назвать **amplification effect**: растет скорость доставки, но одновременно растет скорость масштабирования ошибок.

Поэтому guardrails, инженерные ограничители и автоматические проверки, становятся не «дополнением к зрелому процессу», а базовым условием безопасного AI-ускорения. Иначе команда получает короткий всплеск velocity, а затем платит за него регрессиями, архитектурным дрейфом и дорогим rework.

Практический вывод такой: чем больше доля AI в ежедневной разработке, тем выше должна быть доля автоматизированного контроля на каждом PR и на каждом merge.

## Минимальный набор guardrails

Ниже, минимальный набор, который стоит считать обязательным независимо от стека и размера команды. Конкретные инструменты выбираются под текущий toolchain компании, важно не название инструмента, а роль проверки.

### 1) Linting

Цель linting, сделать код предсказуемым и заранее ловить типовые ошибки и anti-patterns.

Что должно быть обеспечено:
- единый набор правил для репозитория или для домена
- одинаковый результат локально и в CI
- блокировка merge при критичных нарушениях

### 2) Formatting

Форматирование снимает бессмысленные споры на ревью и уменьшает шум в diff.

Что важно:
- автоформат при коммите или перед push
- единые правила во всех сервисах, где это возможно
- отсутствие ручных исключений без явной причины

### 3) Type checking

Статическая проверка типов критична в AI-потоке, потому что AI часто генерирует правдоподобный, но слабо согласованный код на границах модулей.

Что дает type checking:
- раннее обнаружение несовместимых контрактов
- снижение числа runtime-ошибок после merge
- более надежные рефакторинги

### 4) Тесты

Минимум, unit + integration-тесты для критичных сценариев. Не обязательно сразу стремиться к экстремальному coverage, но критические user flows должны быть защищены.

Ключевые принципы:
- тесты обязаны быть стабильными, flaky-тесты подрывают доверие ко всей системе
- AI-изменение без тестового доказательства считается рискованным по умолчанию
- приоритет, проверка бизнес-инвариантов, а не только «счастливого пути»

### 5) CI

CI должен быть единой точкой принятия решения о качестве изменения.

Минимальная последовательность в pipeline:
1. сборка
2. linting + formatting check
3. type check
4. тесты

Если любой шаг падает, merge блокируется. Исключения допускаются редко и фиксируются явно, чтобы не превращаться в «тихую норму».

## Автоматическое принуждение к качеству

Guardrails работают только тогда, когда они обязательны по процессу, а не остаются «рекомендацией». Для этого нужен слой автоматического enforcement.

### Pre-commit hooks

На уровне разработчика hooks позволяют отсеять очевидные нарушения до открытия PR:
- запуск быстрых линтеров
- автоформатирование
- простые структурные проверки

Это снижает шум в CI и экономит время ревьюеров.

### CI checks

В CI должна выполняться полная, воспроизводимая проверка. В отличие от локального окружения, это нейтральная и обязательная среда контроля. Именно она защищает от «у меня локально прошло».

### Structural tests

Часть правил невозможно надежно выразить обычными линтерами. Для архитектурных инвариантов нужны структурные тесты или custom linters. Они проверяют не стиль, а форму системы.

Примеры архитектурных инвариантов:
- слой UI не обращается напрямую к persistence-слою
- доменный слой не зависит от инфраструктурного
- запрещены импорты между bounded context, если нет разрешенного контракта

Такие проверки особенно ценны при AI-генерации, потому что они ловят архитектурный дрейф в момент появления, а не через несколько спринтов.

## Code review для AI-generated code, что добавить к стандартному процессу

Обычное ревью остается обязательным, но для AI-сгенерированных изменений нужен дополнительный чеклист. Цель, не «наказать» разработчика, а закрыть типовые риски машинной генерации.

Минимальные дополнительные требования:

1. **Запрет на `as any` без формального исключения**  
   Любое ослабление типобезопасности должно иметь обоснование и план устранения.

2. **Запрет на пустой `catch`**  
   Ошибки не должны проглатываться. Нужны либо корректная обработка, либо явный проброс, либо логирование по принятому стандарту.

3. **Запрет `console.log` в production-коде**  
   Диагностика должна идти через принятый механизм логирования с контролем уровней и формата.

4. **Проверка соответствия архитектурным ограничениям**  
   Даже «рабочий» код отклоняется, если нарушает границы модулей, слоев и контрактов.

5. **Проверка покрывающих тестов для измененного поведения**  
   Для каждого существенного изменения должен быть тест, который подтверждает новое поведение или защищает от регрессии.

6. **Проверка прозрачности намерения**  
   В PR должно быть ясно, что меняется, почему это делается и какие риски приняты.

Важно, эти требования не привязаны к одному конкретному инструменту. Они совместимы с существующей инфраструктурой команды и могут внедряться поэтапно.

## Как внедрять guardrails без конфликта с существующим toolchain

Сильная ошибка при внедрении guardrails, попытка сразу заменить все инструменты команды «идеальным» стеком. На практике это дает сопротивление и снижает скорость.

Рабочий путь:
- зафиксировать обязательные роли проверок, а не бренды инструментов
- картировать текущий toolchain команды на эти роли
- закрыть пробелы минимальными изменениями
- постепенно усиливать правила после стабилизации базового набора

Так команда сохраняет операционную устойчивость и одновременно повышает качество AI-изменений.

## Decision: Which guardrails to implement first

Если внедрять все сразу невозможно, приоритет должен быть таким:

1. **CI как обязательный gate для merge** с минимальным пайплайном: build, lint/format check, type check, tests.
2. **Pre-commit hooks** для быстрых локальных проверок и автоформата.
3. **Дополнительный AI-review checklist** с явными запретами: `as any` без исключения, пустой `catch`, `console.log` в production.
4. **Structural tests/custom linters** для 2-3 главных архитектурных инвариантов.

Итоговое решение для CTO и engineering leadership: сначала обеспечить обязательный автоматический контроль качества на пути к merge, затем закрепить AI-специфичные правила ревью, после этого масштабировать структурные архитектурные проверки. Такой порядок дает максимальное снижение риска при минимальном организационном трении.

# Development Processes with AI: From Copilot to Agent Fleets

Этот раздел описывает, как меняется процесс разработки при росте роли AI, какие управленческие и инженерные практики нужны на каждом этапе, и какой режим стоит выбрать в первый год внедрения. Ключевая идея простая: скорость растет только там, где есть дисциплина процесса, прозрачная ответственность и ограничение рисков.

## 1) Эволюция режимов работы

Ниже показана практическая эволюция, которую сейчас проходят команды.

### 1. Autocomplete

AI дописывает строки и небольшие блоки кода в IDE. Инициатива полностью у разработчика.

- Роль человека: пишет архитектуру, код и тесты, AI ускоряет рутину.
- Плюсы: низкий риск, быстрый старт, почти не требует перестройки процесса.
- Ограничения: локальная оптимизация, почти нет системного эффекта на delivery.

### 2. Chat

Разработчик обсуждает решение в диалоге, просит фрагменты кода, тесты, объяснения.

- Роль человека: формулирует задачу, проверяет ответы, интегрирует изменения.
- Плюсы: выше скорость исследований, проще разбирать сложные баги.
- Ограничения: качество зависит от контекста, растет риск ошибок при копипасте без ревью.

### 3. Agents

AI получает задачу как рабочий пакет и сам проходит несколько шагов: меняет код, запускает проверки, предлагает итог.

- Роль человека: задает границы, критерии готовности, принимает результат.
- Плюсы: заметный рост пропускной способности команды на типовых изменениях.
- Ограничения: нужен более строгий контроль, иначе растет технический долг.

### 4. Background Agents

Агенты работают асинхронно в фоне, по завершении возвращают pull request или набор изменений.

- Роль человека: приоритизирует очередь задач, проверяет результат, управляет SLA качества.
- Плюсы: команда не блокируется на рутине, лучше используется параллелизм.
- Ограничения: легко потерять видимость контекста, если нет политики ревью и ownership.

### 5. Agent Fleets

Несколько агентов параллельно решают связанные задачи под оркестрацией.

- Роль человека: архитектурный контроль, декомпозиция, контроль интерфейсов и интеграции.
- Плюсы: максимальная скорость в больших бэклогах.
- Ограничения: высокий операционный риск, сложная координация, нужен зрелый engineering management.

Вывод по эволюции: каждый следующий режим добавляет производительность, но одновременно повышает требования к зрелости команды, тестовому контуру и governance.

## 2) Рабочий цикл по Addy Osmani

Практика Addy Osmani хорошо ложится на корпоративную разработку, потому что сочетает высокую скорость и управляемость.

### Шаг 1. Specs before code

Сначала команда фиксирует спецификацию: цель, ограничения, edge cases, критерии приемки. Только после этого начинается генерация кода. Такой порядок снижает объем переделок и делает ответы AI более точными.

### Шаг 2. Small chunks

Реализация дробится на маленькие, независимые шаги. Вместо одного большого запроса команда формирует серию коротких задач. Это упрощает проверку и быстрее выявляет ошибки.

### Шаг 3. Context pack

Перед запуском задачи AI получает пакет контекста: релевантные модули, правила проекта, инварианты, ограничения по производительности и безопасности. Без этого агент начинает гадать, а не решать.

### Шаг 4. Test loop

Каждый шаг замыкается на тестовый цикл: написать или обновить тесты, прогнать проверки, исправить дефекты. AI ускоряет итерации, но не заменяет критерии качества.

### Шаг 5. Commit as checkpoint

После каждого стабильного шага команда делает маленький коммит. Коммиты работают как save points, поэтому откат после неудачной генерации занимает минуты, а не часы.

### Шаг 6. Review

Изменения проходят ревью человеком, даже если код формально прошел тесты. Цель ревью не только найти баг, но и проверить соответствие архитектуре и доменной логике.

Итог: процесс не строится вокруг «попросить AI написать все сразу». Он строится вокруг коротких контролируемых циклов с предсказуемым качеством.

## 3) Модель Harness Engineering: humans steer, agents execute

Подход Harness Engineering формулируется как разделение ролей: люди задают направление и правила, агенты исполняют конкретные задачи.

Практически это выглядит так:

1. Человек формулирует задачу, границы и Definition of Done.
2. Агент исполняет работу внутри этих границ.
3. Человек проверяет результат, принимает, дорабатывает или отклоняет.

Важно, что ответственность не делегируется модели. Ответственность остается у команды. Это позволяет масштабировать разработку без потери управляемости.

Для зрелой команды эта модель дает лучший баланс:

- люди концентрируются на продуктовых решениях и архитектуре,
- агенты берут на себя рутину и повторяемые инженерные операции,
- качество защищается процессом проверки, а не доверием к «умности» модели.

## 4) Практики, которые стоит стандартизировать

### TDD с участием AI

Базовый шаблон: сначала тест, потом реализация, потом рефакторинг. AI можно использовать в каждом шаге, но порядок менять нельзя. Такой режим особенно полезен в командах средней зрелости, где риск регрессий выше.

### Частые коммиты как точки сохранения

Коммит после каждого маленького рабочего шага снижает цену ошибки. Команда быстрее экспериментирует и быстрее откатывается.

### Изоляция задач через git worktrees

Параллельная работа человека и AI в отдельных рабочих деревьях убирает конфликты и упрощает контроль изменений. Это особенно полезно при фоновом режиме и параллельных задачах.

### Multi-model подход

Одна модель генерирует решение, вторая проверяет, третья помогает в анализе дефектов. Такой подход снижает риск систематических ошибок одной модели. При этом процесс должен оставаться tool-agnostic: команда описывает роли моделей, а не привязывается к конкретным вендорам.

## 5) Ревью AI-сгенерированного кода

Обычное ревью здесь недостаточно. Нужен расширенный контур проверки.

### AI-on-AI review

Отдельная модель проверяет изменения на логические дефекты, потенциальные уязвимости, пограничные случаи и неоправданную сложность.

### Дополнительные проверки человеком

Минимальный список:

- соответствие доменным инвариантам,
- обратная совместимость API и схем данных,
- качество обработки ошибок,
- наблюдаемость: логи, метрики, трассировка,
- эксплуатационные риски и влияние на SLO.

Правило: код не мержится, пока reviewer не может объяснить, как решение работает и почему оно безопасно в продакшене.

## 6) Governance: ownership, merge policy, blast radius

### Кто владеет AI-сгенерированным кодом

Владельцем всегда является команда, а не инструмент. За каждый merged change назначается конкретный инженер-владелец, который отвечает за сопровождение.

### Merge policy

Рекомендуемая политика:

1. Обязательные автоматические проверки.
2. Обязательное ревью человеком.
3. Для задач повышенного риска, второе ревью или архитектурное согласование.
4. Явная трассируемость: от требования к изменению и тестам.

### Ограничение blast radius

Нужно ограничивать радиус потенциального ущерба от ошибок AI:

- маленькие PR,
- поэтапный rollout,
- feature flags,
- быстрый rollback,
- запрет на массовые критичные изменения без ручного контроля.

Чем ниже зрелость команды, тем жестче должны быть эти ограничения.

## 7) Учет зрелости команды

Один и тот же режим дает разный результат в командах разного уровня.

- Низкая зрелость: стоит начинать с Autocomplete и Chat, параллельно укрепляя тесты, код-ревью и инженерные стандарты.
- Средняя зрелость: можно переходить к Agents в ограниченных доменах и вводить Background Agents для рутинных задач.
- Высокая зрелость: возможны Agent Fleets на хорошо формализованных потоках работ при сильной архитектурной функции и стабильном governance.

## Decision: Which mode to implement for Year-1

Решение на Year-1: **основной режим внедрения, Agents под управлением человека, плюс ограниченный Background Agents для низкорисковых задач**.

Почему это оптимально:

1. Дает заметный прирост скорости по сравнению с Autocomplete и Chat.
2. Сохраняет управляемость через четкий human approval перед merge.
3. Не требует сразу переходить к сложной оркестрации уровня Agent Fleets.
4. Позволяет поэтапно нарастить зрелость процессов, тестов и governance.

Что откладываем:

- Полноценные Agent Fleets переносим на следующий этап, после накопления метрик качества, инцидентов и эффективности за Year-1.

Критерий успешности Year-1:

- рост инженерной производительности без роста дефектов в продакшене,
- предсказуемый cycle time,
- прозрачная ответственность за каждый AI-сгенерированный change.

# Team Training and New Roles

Переход к AI-ready разработке меняет не только инструменты, но и саму роль инженерной команды. Если раньше главный вопрос был «кто быстрее пишет код», то сейчас важнее другой, кто проектирует среду, в которой агент пишет код предсказуемо, безопасно и в нужном архитектурном стиле.

Практика команд, которые уже работают в таком режиме, показывает одну закономерность. Скорость растет не из-за того, что модель «умнее всех», а из-за того, что команда перестраивает роли, правила и обучение вокруг новой реальности. В материалах про harness engineering этот сдвиг описывается очень прямо: узкое место находится не в генерации кода, а в среде вокруг агента, в границах, проверках, документации и качестве ревью.

## Почему роли нужно пересобрать сейчас

Для нашей текущей структуры это не означает «сломать все и набрать новый отдел». Наоборот, изменения нужно встроить в существующую команду, где уже есть инженеры, техлиды, архитекторы и инженерные менеджеры.

Новая модель работы добавляет к привычным ролям три специализированных контура:

1. Контур проектирования среды для агентов.
2. Контур проверки AI-сгенерированных изменений.
3. Контур масштабирования практик внутри команд.

Из этих контуров формируются новые роли: Harness Engineer, AI Reviewer и AI Champion.

## Новые роли в существующей инженерной структуре

### 1) Harness Engineer

Главный переход в инженерной профессии выглядит так: **от «писать код руками» к «проектировать среду, где агент пишет правильный код»**.

Harness Engineer отвечает за рабочий контур агента:

- структуру проекта, которую агент может надежно читать;
- guardrails в линтерах, тестах и CI;
- AGENTS.md, инструкции и примеры «как у нас принято делать»;
- доступ агента к нужным внутренним инструментам через понятные интерфейсы;
- быстрые обратные связи, чтобы типовые ошибки фиксировались один раз и больше не повторялись.

Ключевая идея роли простая. Если агент ошибся, проблема считается не только «ошибкой модели», но и сигналом, что среда была недоописана или недозащищена.

### 2) AI Reviewer

AI Reviewer это инженер, который специализируется на проверке кода, подготовленного агентами, и держит качество на уровне не ниже ручной разработки.

Его зона ответственности:

- проверка архитектурной корректности, а не только синтаксиса;
- поиск скрытых дефектов, ложных допущений и хрупких абстракций;
- контроль семантики изменений, backward compatibility и рисков поддержки;
- проверка того, что код соответствует внутренним паттернам, а не случайному стилю из ответа модели.

Важно, что AI Reviewer не «тормозит delivery», а защищает скорость на длинной дистанции. Без такой роли команда быстро получит рост объема PR, но через 2-3 квартала упрется в техдолг и каскадные регрессии.

### 3) AI Champion

AI Champion нужен для распространения практик между командами. Это связующее звено между платформенными решениями и ежедневной работой инженеров.

Его задача:

- запускать и поддерживать внутреннюю сеть чемпионов;
- помогать командам внедрять рабочие паттерны в реальных задачах;
- собирать обратную связь, где агент реально помогает, а где мешает;
- поднимать общий уровень навыков через примеры, разборы и наставничество.

Без AI Champion внедрение почти всегда становится неравномерным. Часть команд ускоряется, часть остается в старом процессе, и организация получает фрагментированный ландшафт.

## Архитектор как дирижер

В AI-ready модели роль архитектора не уменьшается, а усиливается. Архитектор становится дирижером, который держит целостность системы при высокой скорости генерации изменений.

Его фокус:

- **Guardrails:** четкие ограничения по слоям, зависимостям, naming и интеграционным точкам.
- **Patterns:** библиотека одобренных архитектурных и кодовых шаблонов для типовых задач.
- **Boundaries:** явные границы модулей и сервисов, которые проверяются автоматически.

Это не «бюрократия ради процесса». Для агента такие ограничения работают как карта местности. Чем лучше карта, тем меньше случайных решений, меньше мусорных правок и выше предсказуемость результата.

## Новые компетенции, которые нужно развивать

Чтобы роли заработали, команде нужно целевое развитие навыков. Базовый набор включает три группы.

### Prompt engineering

Навык постановки задачи агенту в понятной, проверяемой форме:

- формулировать цель, ограничения и критерии готовности;
- дробить большую задачу на проверяемые шаги;
- задавать контрольные вопросы модели до начала генерации кода;
- требовать план и альтернативы, а не только финальный патч.

### Context engineering

Навык управления контекстом, который получает агент:

- правильный выбор файлов и артефактов для загрузки;
- явная передача архитектурных решений и бизнес-ограничений;
- поддержка AGENTS.md и локальных инструкций в актуальном виде;
- проектирование структуры репозитория так, чтобы агент видел цельный сценарий, а не разрозненные куски.

### AI review skills

Навык профессиональной проверки AI-сгенерированного кода:

- выявлять «правдоподобные, но неверные» решения;
- находить деградацию читаемости и будущей сопровождаемости;
- проверять тестовые пробелы и ложноположительное «все прошло»;
- оценивать риски эксплуатации и поддержки до merge.

## Риски человеческого фактора

Технические изменения обычно видны сразу, организационные риски проявляются позже. Для текущего этапа особенно критичны три сценария.

### 1) Джуны начинают слишком доверять AI

Симптомы:

- принятие ответа модели без проверки гипотез;
- слабое понимание, почему код работает;
- перенос ответственности на инструмент.

Риск: накопление скрытых ошибок и просадка инженерного мышления.

Контрмера: обязательное ревью reasoning и короткие разборы «почему это решение верное или неверное», а не только проверка тестов.

### 2) Сеньоры игнорируют AI

Симптомы:

- демонстративный отказ менять привычный процесс;
- скепсис без экспериментов на реальных задачах;
- потеря роли наставников в новой модели.

Риск: замедление команды и разрыв между лидерами и практикой разработки.

Контрмера: фокус на пилотах с измеримым результатом, где опытные инженеры участвуют как архитектурные владельцы и AI Reviewer, а не как «сторонние наблюдатели».

### 3) Неравномерное внедрение между командами

Симптомы:

- у одних команд высокая скорость и стабильный quality, у других хаос;
- разные стандарты промптов, ревью и CI;
- конфликт ожиданий между продуктом, платформой и delivery-командами.

Риск: организационная фрагментация, когда общий процесс перестает быть общим.

Контрмера: сеть AI Champion, единые минимальные стандарты и прозрачные метрики adoption по всем командам.

## Контур обучения: формат, без перегруза и академизма

Нам нужна не «длинная учебная программа на много недель», а прикладной контур, встроенный в рабочий ритм команды.

### Компонент 1. Champions network

В каждом инженерном домене назначаются AI Champions. Это не отдельный департамент, а дополнительные обязанности у сильных практиков.

Что делает сеть:

- помогает командам перейти на единые минимальные практики;
- проводит короткие внедренческие сессии по реальным кейсам;
- эскалирует системные проблемы в платформенную и архитектурную функцию.

### Компонент 2. Практические воркшопы

Формат воркшопа:

- одна реальная инженерная задача;
- сначала план с агентом, потом реализация;
- обязательный разбор рисков и ошибок;
- итогом становится не «слайды», а примененный в репозитории паттерн.

Цель воркшопов, быстро поднять общий уровень Prompt engineering, Context engineering и AI review на примерах из нашего кода.

### Компонент 3. Pair programming with AI

Парная работа строится в формате «инженер + агент + reviewer». Для джунов это дает безопасный способ учиться, для сеньоров это ускоряет переход от индивидуальной практики к командному стандарту.

Правило простое:

- агент предлагает варианты;
- инженер принимает архитектурные решения;
- reviewer фиксирует качество и соответствие guardrails.

Такой режим снижает риск слепого доверия к AI и одновременно не превращает процесс в ручное переписывание каждого ответа модели.

## Как встроить это в действующую структуру команды

Чтобы не ломать организацию, роли накладываются на текущий каркас:

- **Architect / Tech Lead** берет функцию «дирижера», владеет guardrails и паттернами.
- **Senior Engineer** частично берет роль AI Reviewer для критичных доменов.
- **Selected Engineers в командах** получают роль AI Champion и поддерживают adoption локально.
- **Platform/DevEx** помогает Harness Engineer с инструментами, CI и политиками.

Такой подход сохраняет существующую иерархию ответственности, но добавляет недостающие функции для работы с агентами.

## Метрики эффективности обучения

Чтобы программа не осталась декларацией, нужны короткие измеримые индикаторы:

- доля PR с AI-участием и доля PR, успешно прошедших с первого раза;
- число повторяющихся ошибок агента до и после обновления guardrails;
- скорость онбординга нового инженера до первого полезного merge;
- доля команд, где действуют единые стандарты контекста и ревью;
- динамика регрессий на критичных пользовательских сценариях.

Если показатели улучшаются без роста инцидентов, программа работает правильно.

## Decision: Training format and timeline, who is responsible

**Решение:** принять формат обучения как встроенный производственный контур из трех элементов, champions network, практические воркшопы, pair programming with AI. Запускать поэтапно, через короткий пилот и масштабирование на все инженерные команды после проверки метрик.

**Таймлайн:**

- **Этап 1, Pilot:** выбрать 2 команды, назначить роли Harness Engineer, AI Reviewer, AI Champion, провести первые практические сессии и зафиксировать базовые метрики.
- **Этап 2, Scale:** расширить модель на остальные команды, унифицировать guardrails и формат AI review.
- **Этап 3, Institutionalize:** закрепить роли и практики в стандартном инженерном процессе, включая онбординг и регулярные архитектурные ревизии.

**Ответственность:**

- **CTO / VP Engineering:** утверждает рамку и целевые метрики.
- **Architecture function:** владеет guardrails, паттернами и границами.
- **Engineering Managers:** обеспечивают внедрение в командах и контроль ритма обучения.
- **AI Champions network:** ведет локальную адаптацию и обмен практиками.
- **AI Reviewers:** поддерживают бар качества для AI-сгенерированного кода.

Финальный принцип: мы не обучаем команду «пользоваться модным инструментом». Мы строим новую инженерную операционную модель, где люди сохраняют ответственность за архитектуру и качество, а агенты дают ускорение без потери управляемости.

# Implementation Roadmap (4 Phases)

Этот раздел фиксирует практический план внедрения AI-ассистированной разработки в 4 фазы. Цель плана: снизить операционный риск и дать управляемый путь от текущего состояния к масштабированному применению инструментов в продуктовых командах.

План построен так, чтобы каждая следующая фаза стартовала только после подтвержденных результатов предыдущей. Это важно по двум причинам. Во-первых, без базовой диагностики нельзя честно оценить эффект от пилота. Во-вторых, без технической гигиены и организационных правил пилот почти всегда дает локальный успех, который не переносится на другие команды.

## Принципы выполнения дорожной карты

1. **Измеримость до автоматизации**. Сначала фиксируем исходные метрики, потом внедряем практики и инструменты.
2. **Ограниченный пилот вместо широкого запуска**. Один модуль и одна команда дают быстрый цикл обратной связи.
3. **Безопасность и контроль доступа по умолчанию**. Никаких ad-hoc интеграций без политики секретов, логирования и ролевой модели.
4. **Стандартизация через реальные кейсы**. Общие стандарты формируем на основании пилота, а не теоретических предположений.
5. **Непрерывное улучшение**. После масштабирования метрики продолжают измеряться в постоянном режиме.

## Фаза 0. Audit (1-2 weeks)

### Цель фазы

Сформировать объективную исходную точку: понять текущее состояние кодовой базы, процессов поставки и инженерной культуры, чтобы далее измерять эффект внедрения AI-практик на сопоставимой основе.

### Критерии входа

- Подтвержден спонсор инициативы на уровне руководителя инженерной функции или CTO.
- Назначен владелец программы внедрения.
- Есть доступ к репозиториям, CI/CD, баг-трекеру и системам наблюдаемости.
- Зафиксирован периметр первичного аудита, продукт, команды, критичные сервисы.

### Критерии выхода

- Проведен аудит архитектуры и состояния кодовой базы по ключевым модулям.
- Собраны baseline-метрики: DORA, cycle time, defect rate, test coverage.
- Определен текущий уровень **AI Readiness Maturity** по согласованной модели оценки.
- Выбрана пилотная команда и пилотный модуль с обоснованием.
- Подготовлен краткий аудит-отчет с приоритетами на следующие 2 фазы.

### Ответственные роли

- **CTO / Head of Engineering**: утверждение рамок, целей и критериев успеха.
- **Program Lead (AI Enablement Lead)**: координация аудита, синхронизация ролей.
- **Tech Leads / Architects**: технический анализ архитектуры и качества кода.
- **DevOps / Platform Engineer**: сбор данных CI/CD и DORA.
- **QA Lead**: метрики дефектов и качества тестового покрытия.
- **Security Lead**: ограничения по данным, доступам и требованиям комплаенса.

### Конкретные действия

1. **Диагностика кодовой базы**
   - Оценить размер и структуру основных репозиториев.
   - Выявить зоны высокой связности, дублирования и сложной поддержки.
   - Определить модули с наибольшей частотой изменений и инцидентов.

2. **Сбор baseline-метрик за репрезентативный период**
   - DORA: deployment frequency, lead time for changes, change failure rate, time to restore.
   - Cycle time: от старта задачи до продакшн-деплоя.
   - Defect rate: дефекты на релиз, доля регрессий, дефекты после релиза.
   - Test coverage: модульное, интеграционное, e2e, включая стабильность прогона.

3. **Оценка AI Readiness Maturity**
   - Измерить зрелость по осям: код, процессы, документация, guardrails, обучение.
   - Присвоить текущий уровень зрелости и зафиксировать разрыв до целевого.

4. **Выбор пилота**
   - Сформировать короткий список команд/модулей.
   - Сравнить кандидатов по критериям: бизнес-влияние, риск, автономность команды, наблюдаемость метрик.
   - Утвердить 1 пилотную команду и 1 пилотный модуль.

### Основные риски и меры

- **Риск: неполные или несопоставимые метрики**.
  - Мера: единый период измерения и единые определения всех KPI.
- **Риск: политический выбор пилота вместо рационального**.
  - Мера: прозрачная матрица критериев с весами и письменным обоснованием.
- **Риск: недооценка ограничений безопасности**.
  - Мера: обязательная проверка Security Lead до финального решения по пилоту.

## Фаза 1. Cleanup and guardrails (2-4 weeks)

### Цель фазы

Подготовить техническую и процессную базу, чтобы пилот шел в контролируемой среде. На этом этапе устраняется критичный технический долг, вводятся минимальные инженерные стандарты и создаются рабочие правила взаимодействия человека и AI.

### Критерии входа

- Завершена фаза 0, утвержден audit-отчет.
- Выбраны пилотная команда и пилотный модуль.
- Согласованы минимальные quality gates для пилотного репозитория.

### Критерии выхода

- Снижены ключевые точки дублирования в пилотном модуле.
- Настроены CI, linting, formatting с обязательной проверкой в pull request.
- Создан и согласован файл `AGENTS.md` с правилами работы AI-агентов.
- Задокументирована архитектура пилотного модуля и контекст зависимостей.
- Промаркированы legacy-зоны, где AI-предложения требуют ручной эскалации.

### Ответственные роли

- **Tech Lead пилотной команды**: владелец технических решений по подготовке.
- **Senior Developers**: рефакторинг, устранение дублирования, codemods.
- **DevOps / Platform**: CI пайплайны, quality gates, контроль статусов.
- **QA Engineer**: проверка валидности тестовых ворот.
- **Security Lead**: ревью guardrails и ограничений.
- **Engineering Manager**: приоритизация и защита фокуса команды.

### Конкретные действия

1. **Очистка пилотного периметра**
   - Убрать очевидное дублирование в коде, утилитах и конфигурации.
   - Выровнять структуру директорий и соглашения по именованию.
   - Устранить нестабильные тесты, которые создают шум в CI.

2. **Введение технических guardrails**
   - Подключить линтеры и форматтеры как обязательный шаг проверки.
   - Настроить минимальный порог тестового покрытия на изменяемый код.
   - Зафиксировать требования к PR: описание задачи, план проверки, риски.

3. **Документирование и рабочие инструкции**
   - Подготовить `AGENTS.md`: допустимые задачи для AI, правила промптинга, ограничения по данным.
   - Описать архитектурный контур модуля: границы, контракты, критичные зависимости.
   - Явно отметить legacy-зоны, где изменения требуют отдельного согласования.

4. **Проверка готовности к пилоту**
   - Провести dry-run сценариев: изменение фичи, багфикс, добавление тестов.
   - Убедиться, что пайплайн стабилен, а команды понимают новые правила.

### Основные риски и меры

- **Риск: команда воспринимает фазу как бюрократию**.
  - Мера: показывать связь каждого правила с реальными инцидентами и потерями времени.
- **Риск: guardrails слишком жесткие и замедляют разработку**.
  - Мера: задать минимальный рабочий набор, затем усиливать по мере стабилизации.
- **Риск: документация быстро устаревает**.
  - Мера: ввести ответственность за обновление документов в definition of done.

## Фаза 2. Pilot (4-8 weeks)

### Цель фазы

Проверить на практике, дает ли AI-ассистированная разработка измеримый эффект в выбранном модуле, при сохранении качества, предсказуемости поставки и требований безопасности.

### Критерии входа

- Завершена фаза 1, технические и процессные guardrails активны.
- У пилотной команды есть доступ к одобренному self-hosted инструментарию.
- Назначены метрики успеха пилота и частота их обзора.

### Критерии выхода

- Развернут и стабилизирован набор self-hosted AI-инструментов для пилота.
- Команда прошла обучение и демонстрирует применение новых практик в реальных задачах.
- Собраны метрики по пилотному периоду и проведено сравнение с baseline.
- Зафиксирован список рабочих практик, анти-паттернов и корректировок.
- Подготовлено решение о масштабировании с оценкой эффекта и рисков.

### Ответственные роли

- **Pilot Tech Lead**: владелец технического результата пилота.
- **Platform / MLOps Engineer**: развертывание и поддержка self-hosted инструментов.
- **Developers пилотной команды**: применение практик в ежедневной работе.
- **QA Lead**: контроль влияния на качество и регрессии.
- **Security / Compliance**: контроль соблюдения ограничений по данным.
- **Program Lead**: еженедельный обзор прогресса и снятие блокеров.

### Конкретные действия

1. **Развертывание self-hosted инструментов**
   - Поднять пилотный контур с доступом по ролям и аудитом действий.
   - Интегрировать инструменты в IDE, code review и внутренние workflow.
   - Зафиксировать fallback-процедуры на случай недоступности сервиса.

2. **Обучение пилотной команды**
   - Провести практические сессии по типовым сценариям: рефакторинг, написание тестов, анализ PR.
   - Ввести правила безопасной работы с кодом и контекстом.
   - Назначить внутренних наставников по ролям.

3. **Исполнение пилота в боевом режиме**
   - Пропустить через новые практики поток реальных задач.
   - Проводить еженедельный разбор: где AI ускоряет, где увеличивает риск.
   - Исправлять process gaps сразу, не откладывая до конца фазы.

4. **Сравнение с baseline и корректировка**
   - Сопоставить DORA, cycle time, defect rate, test coverage до и после пилота.
   - Разделить эффект на короткий и устойчивый, чтобы не путать адаптацию с долгосрочным улучшением.
   - Обновить `AGENTS.md`, чеклисты и стандарты на основе фактических результатов.

### Основные риски и меры

- **Риск: эффект пилота завышен из-за фокуса и ручного сопровождения**.
  - Мера: фиксировать затраты времени на поддержку и учитывать их в итоговой оценке.
- **Риск: рост дефектов при ускорении разработки**.
  - Мера: не ослаблять quality gates, усиливать тестовые сценарии на зоне изменений.
- **Риск: инструмент используется неравномерно внутри команды**.
  - Мера: ролевые наставники, парные сессии, короткие еженедельные ретро.

## Фаза 3. Scaling (8-12 weeks)

### Цель фазы

Перенести проверенные практики из пилота в остальные команды, сохранив качество, безопасность и управляемость. На выходе организация должна получить единый рабочий стандарт AI-ассистированной разработки.

### Критерии входа

- Пилот завершен, его эффект подтвержден метриками.
- Утвержден набор практик, которые показали стабильный результат.
- Есть план поэтапного подключения команд и ресурсная модель поддержки.

### Критерии выхода

- Практики внедрены во всех целевых командах в согласованном объеме.
- Действует сеть champions, которая помогает командам в адаптации.
- Метрики качества и скорости собираются непрерывно и обсуждаются регулярно.
- Создан цикл постоянного улучшения стандартов, документации и обучения.

### Ответственные роли

- **Head of Engineering / CTO**: стратегическое управление масштабированием.
- **Program Lead**: операционное выполнение плана по волнам внедрения.
- **AI Champions (по командам)**: локальная поддержка и разбор сложных кейсов.
- **Platform Team**: доступность платформы, производительность, SLA.
- **Security / Compliance**: периодический контроль соблюдения политик.
- **People / Learning Partner**: системное обучение и онбординг новых сотрудников.

### Конкретные действия

1. **Расширение по волнам**
   - Подключать команды партиями, начиная с наиболее близких к пилотному профилю.
   - Для каждой волны запускать короткий readiness-чек по единым критериям.
   - Не переводить следующую волну, пока текущая не стабилизирована.

2. **Стандартизация практик**
   - Утвердить единый набор шаблонов: PR, issue, тестовые чеклисты, правила для AI.
   - Поддерживать централизованный `AGENTS.md` и локальные дополнения команд.
   - Обновить инженерные регламенты и критерии done.

3. **Сеть champions**
   - Назначить champion в каждой команде.
   - Проводить регулярные встречи champions для обмена кейсами.
   - Поддерживать внутреннюю базу рабочих паттернов и типовых ошибок.

4. **Непрерывное измерение и улучшение**
   - Собирать метрики на уровне команды, продукта и организации.
   - Проводить ежемесячный обзор KPI и проблемных зон.
   - Корректировать практики, обучение и инструменты на основе данных.

### Основные риски и меры

- **Риск: потеря качества при ускоренном расширении**.
  - Мера: волновое подключение, обязательный readiness-чек и пост-проверка качества.
- **Риск: фрагментация стандартов между командами**.
  - Мера: централизованные шаблоны, единый каталог практик и governance-ритм.
- **Риск: усталость от изменений**.
  - Мера: короткие итерации, прозрачная коммуникация эффекта, реальная помощь champions.

## Единый формат контроля по фазам

Чтобы план не превратился в формальность, каждую фазу закрывает короткий go/no-go обзор. Рекомендуемый формат:

- Что планировали получить в фазе.
- Что фактически получили по метрикам и артефактам.
- Какие риски остаются и как они закрываются в следующей фазе.
- Решение: продолжать, скорректировать, остановить.

Этот формат дисциплинирует внедрение и защищает организацию от двух крайностей: слишком раннего масштабирования и чрезмерно затянутого пилота.

## Decision: Which team/module to start pilot with, timeline

**Решение по старту пилота**:

- **Пилотная команда**: команда Core Product Backend.
- **Пилотный модуль**: модуль `Order Management API`.

**Почему именно этот выбор**:

1. Модуль дает высокий бизнес-эффект, но не относится к самым критичным зонам с максимальным риском для непрерывности бизнеса.
2. Граница модуля достаточно четкая, можно изолировать эффект и корректно измерять метрики.
3. Команда работает в регулярном релизном ритме, поэтому сравнение baseline и результата пилота будет статистически устойчивым.
4. Технический профиль модуля типичен для других backend-команд, что повышает переносимость практик в фазе масштабирования.

**Таймлайн внедрения**:

- Недели 1-2: Фаза 0, Audit.
- Недели 3-6: Фаза 1, Cleanup and guardrails.
- Недели 7-14: Фаза 2, Pilot.
- Недели 15-26: Фаза 3, Scaling.

Итоговая длительность программы: 26 недель. После завершения 26-й недели выполняется итоговый обзор на уровне инженерного руководства с подтверждением постоянного режима измерений и улучшений.

# 11. Metrics and KPI

Внедрение AI в инженерные процессы почти всегда начинается с сильного эффекта на уровне ощущений. Команда говорит, что код пишется быстрее, ревью проходит легче, документация появляется без боли. Это хороший сигнал, но для управления программой трансформации его недостаточно. Если не договориться о метриках заранее, через 2-3 месяца обсуждение упрется в субъективные мнения, а не в данные.

Для устойчивого результата нужна трехуровневая модель оценки:

1. **Input-метрики (adoption)**: насколько широко и регулярно команда использует инструменты.
2. **Output-метрики (productivity)**: как меняется производственный поток разработки.
3. **Outcome-метрики (business)**: как это влияет на продукт и бизнес-результат.

Главный принцип раздела: **сначала фиксируем baseline, потом сравниваем динамику**. Без baseline любые «улучшения» нельзя корректно доказать.

## 11.1. Базовые правила измерения

Перед запуском инициативы нужно зафиксировать единые правила.

### Правило 1. Baseline обязателен

Для каждой метрики фиксируется исходное значение за период **до внедрения** (обычно 6-12 недель). Это обязательное условие для всех уровней Input, Output и Outcome.

### Правило 2. Одна метрика, один владелец

У каждой метрики есть owner: инженерный менеджер, product manager, QA lead, platform lead или аналитик. Если owner не определен, метрика быстро теряет актуальность.

### Правило 3. Сравнимые окна наблюдения

Нельзя сравнивать разные сезоны релизов, разные типы команд или разные масштабы задач без поправок. Минимум, что нужно делать, это сравнение одинаковых интервалов и нормализация по типу команды.

### Правило 4. Метрики не должны ломать поведение

Если команда начинает «играть» в цифры, программа деградирует. Поэтому каждый KPI должен иметь анти-метрику или проверку качества. Пример: рост скорости PR проверяется дефектами после релиза.

## 11.2. Input-метрики: Adoption

Input-метрики показывают, насколько программа реально принята разработчиками, а не только закуплена на уровне лицензий.

### 1) Adoption rate

**Что измеряем:** долю инженеров, которые используют AI-инструмент минимум N раз в неделю.

**Формула:**

`Adoption Rate = Кол-во активных пользователей / Кол-во целевых пользователей * 100%`

**Как собирать:**
- логи платформы AI-ассистента,
- SSO-отчеты по активным сессиям,
- данные по командам из HRIS.

**Baseline (до внедрения):** 0% для нового инструмента или фактический уровень «теневого» использования, если команда уже пробовала AI неформально.

**Целевой ориентир по фазам:**
- Пилот: 30-50%
- Масштабирование: 60-75%
- Стандартизация: 80%+

### 2) Usage frequency

**Что измеряем:** как часто активный инженер обращается к AI за период.

**Формула:**

`Usage Frequency = Кол-во AI-взаимодействий / Кол-во активных пользователей`

Можно считать в разрезе день, неделя, спринт.

**Как собирать:** события prompt/request из журналов инструмента, сгруппированные по пользователю и периоду.

**Baseline:** частота использования существующих средств ускорения разработки без AI, например шаблонов и внутренних генераторов.

**Пояснение:** высокая adoption при низкой frequency может означать формальное принятие, но не реальную интеграцию в ежедневный поток.

### 3) % AI-assisted PRs

**Что измеряем:** долю Pull Request, где AI использовался для генерации кода, тестов, описаний, рефакторинга или анализа.

**Формула:**

`% AI-assisted PRs = PR с AI-участием / Все PR * 100%`

**Как собирать:**
- маркеры в шаблоне PR (checkbox),
- тег в комментарии бота,
- событие из CI-плагина.

**Baseline:** 0% или текущая доля по ручной самооценке за 2-4 недели до запуска.

**Риск и контроль:** самоотчет может искажаться. Нужна валидация по техническим следам, например по событиям расширения IDE.

## 11.3. Output-метрики: Productivity

Output-метрики показывают, как меняется сам процесс разработки. Здесь важно измерять не только скорость, но и стабильность, качество и долю переделок.

### 1) PR velocity

**Что измеряем:** сколько PR проходит через систему за период и как быстро они доходят до merge.

Подходы:
- PR merged per engineer per week,
- медианное время от открытия PR до merge.

**Как собирать:** GitHub/GitLab API, данные по статусам PR, время создания и merge.

**Baseline:** 6-12 недель до пилота, отдельно по командам и типам сервисов.

**Контроль качества:** рост velocity сверяется с defect rate и rework rate.

### 2) Cycle time

**Что измеряем:** полное время от старта работы по задаче до выхода изменения в production.

Минимальный разрез:
- In Progress -> Code Review,
- Code Review -> Merge,
- Merge -> Deploy.

**Как собирать:** связка Jira, Git, CI/CD.

**Baseline:** медиана и 75-й перцентиль за период до внедрения.

**Почему важно:** часто AI ускоряет кодирование, но bottleneck остается в review или тестах. Cycle time показывает реальную картину потока.

### 3) Test coverage

**Что измеряем:** долю кода, покрытого автотестами, и динамику покрытия после внедрения AI.

**Как собирать:** отчеты покрытия из CI (JaCoCo, Istanbul, coverage.py и другие).

**Baseline:** текущее покрытие по репозиториям и критическим модулям.

**Комментарий:** здесь важны не только проценты. Рекомендуется добавлять метрику стабильности тестов, чтобы исключить «хрупкие» тесты ради цифры.

### 4) Rework rate

**Что измеряем:** долю изменений, которые пришлось переделывать после первичного merge из-за дефектов, пропущенных требований или некачественной реализации.

**Формула (пример):**

`Rework Rate = Задачи/PR с повторной доработкой / Все завершенные задачи/PR * 100%`

**Как собирать:**
- теги в Jira (reopen, bugfix-after-release),
- повторные PR к той же задаче,
- hotfix связанный с недавним релизом.

**Baseline:** исторический уровень за последний квартал до внедрения.

**Зачем нужно:** это защита от ложного роста скорости. Если velocity растет, а rework растет еще быстрее, программа внедрения требует корректировки.

## 11.4. Outcome-метрики: Business

Outcome-слой отвечает на вопрос, приносит ли AI-инициатива ценность для продукта и бизнеса.

### 1) Time-to-market

**Что измеряем:** время от согласования фичи до доступности для конечного пользователя.

**Как собирать:** roadmap-система + релизные данные + аналитика выпуска.

**Baseline:** среднее и медианное время для сопоставимых типов фич до запуска AI-программы.

**Практика:** оценивать отдельно для small, medium, large инициатив. Иначе одна крупная программа «перекрывает» картину.

### 2) Defect rate

**Что измеряем:** частота дефектов после релиза на единицу изменения.

Примеры нормализации:
- дефекты на 100 story points,
- дефекты на 1000 строк измененного кода,
- дефекты на релиз.

**Как собирать:** баг-трекер, прод-инциденты, postmortem-данные.

**Baseline:** минимум один квартал до внедрения, лучше два.

### 3) DORA metrics

Минимальный набор:
- **Deployment Frequency**,
- **Lead Time for Changes**,
- **Change Failure Rate**,
- **Mean Time to Restore (MTTR)**.

**Как собирать:** CI/CD, мониторинг инцидентов, incident management.

**Baseline:** исторические показатели по каждой команде до AI.

**Комментарий:** DORA дает хорошую опорную рамку для сравнения зрелости команд и позволяет обсуждать изменения на языке engineering leadership.

### 4) Developer satisfaction

**Что измеряем:** восприятие полезности AI, влияние на когнитивную нагрузку, ощущение контроля над кодом и качество рабочего дня.

**Как собирать:** короткий ежемесячный опрос (5-7 вопросов), eNPS для инженерной группы, фокус-группы.

**Baseline:** стартовый опрос до запуска пилота.

**Почему критично:** при одинаковых цифрах продуктивности команда с низкой удовлетворенностью чаще демонстрирует отток, сопротивление и скрытую деградацию качества.

### Рекомендации по отчетности

1. **Операционный ритм:**
   - еженедельный обзор Input + Output,
   - ежемесячный обзор Outcome,
   - квартальный пересмотр целей и порогов.

2. **Единый формат визуализации:**
   - baseline,
   - текущее значение,
   - delta к baseline,
   - статус (green/yellow/red),
   - короткий комментарий владельца.

3. **Контекст обязателен:**
   Любая цифра показывается с пояснением: изменение состава команды, релизный пик, миграции, крупные инциденты.

## 11.6. Фазы внедрения и управленческие пороги

Чтобы метрики помогали принимать решения, для каждой фазы задаются критерии входа и выхода.

### Фаза 1. Пилот

**Цель:** доказать, что инструмент применяется и дает локальный прирост.

Фокус метрик:
- Adoption rate,
- Usage frequency,
- % AI-assisted PRs,
- PR velocity,
- ранние сигналы rework.

Порог перехода в фазу 2:
- стабильный adoption,
- положительный сдвиг velocity или cycle time,
- нет ухудшения defect rate на пилотной выборке.

### Фаза 2. Масштабирование

**Цель:** масштабировать практики без потери качества.

Фокус метрик:
- все Input,
- полный набор Output,
- первые Outcome (time-to-market, defect rate).

Порог перехода в фазу 3:
- улучшения подтверждаются минимум на 2-3 командах,
- rework и defect rate под контролем,
- управляемая стоимость владения платформой.

### Фаза 3. Стандартизация

**Цель:** сделать AI частью операционной модели engineering-организации.

Фокус метрик:
- DORA в регулярной отчетности,
- time-to-market на уровне портфеля,
- developer satisfaction как индикатор устойчивости,
- экономический эффект и прогноз емкости.

Критерий зрелости:
- метрики встроены в регулярный ритм управления,
- есть прозрачные владельцы,
- baseline и динамика поддерживаются автоматически.

## 11.7. Типичные ошибки в KPI-модели

1. **Смотреть только на adoption.**
   Если использование растет, а outcome не улучшается, программа не достигла цели.

2. **Оценивать скорость без качества.**
   Быстрый merge при высоком rework и defect rate означает перенос затрат в будущее.

3. **Отсутствие baseline.**
   Без исходной точки нельзя доказать ни успех, ни провал.

4. **Смешивать команды без сегментации.**
   Платформенная команда и продуктовая команда имеют разную структуру задач. Их нельзя сравнивать в лоб.

5. **Редкая отчетность.**
   При квартальном обновлении часть сигналов приходит слишком поздно.

## 11.8. Decision: Which metrics to implement at each phase

**Phase 1 (Pilot, первые 6-10 недель):**
- Обязательно: Adoption rate, Usage frequency, % AI-assisted PRs, PR velocity, baseline defect snapshot.
- Задача фазы: подтвердить реальное использование и ранний прирост производительности без деградации качества.

**Phase 2 (Scale, следующие 2-3 квартала):**
- Обязательно: полный Input + Output набор, Time-to-market, Defect rate.
- Задача фазы: доказать воспроизводимость результата на нескольких командах и удержать качество на масштабе.

**Phase 3 (Standardize, постоянный контур):**
- Обязательно: DORA metrics, Time-to-market на портфельном уровне, Developer satisfaction, регулярное обновление baseline для новых доменов.
- Задача фазы: перевести AI из эксперимента в системную инженерную практику с прогнозируемым бизнес-эффектом.

# Риски и меры снижения при переходе к AI-ready разработке

## Введение

Переход к AI-ready модели ускоряет разработку, но вместе со скоростью приносит новые классы рисков. Ошибка в таком контуре часто масштабируется быстрее, чем в классическом процессе, потому что модель может генерировать большой объем изменений за короткое время. Это меняет сам профиль управления качеством, безопасностью и ответственностью.

В этом разделе зафиксирован практический реестр рисков для Year-1. Для каждого риска указаны четыре обязательных элемента: описание, вероятность, влияние и меры снижения. Цель не в том, чтобы убрать все риски, это невозможно и экономически невыгодно. Цель в другом, заранее определить, какие риски мы принимаем осознанно, а какие снижаем в первую очередь до начала масштабирования.

Ниже используется следующая шкала:

- **Вероятность:** низкая, средняя, высокая.
- **Влияние:** низкое, среднее, высокое, критическое.

## Реестр рисков

### Риск 1. Усиление технического долга

**Описание.**
AI ускоряет доставку фич, но без строгих архитектурных ограничений генерирует локально работающие, слабо согласованные решения. На короткой дистанции это выглядит как рост productivity, на средней дистанции команда получает рост связности между модулями, усложнение тестирования, дублирование логики и деградацию читаемости. Отдельная проблема, накопление «тихого долга», когда код проходит CI, но ухудшает поддержку.

**Вероятность:** высокая.

**Влияние:** высокое.

**Меры снижения.**

1. Ввести жесткие архитектурные guardrails, правила слоев, зависимостей и публичных интерфейсов.
2. Ограничить размер PR и требовать декомпозицию больших AI-изменений на малые шаги.
3. Сделать обязательным AI review + human review для всех non-trivial изменений.
4. Включить метрики техдолга в регулярный ритм, churn, сложность, дублирование, доля исправлений после merge.
5. Закрепить владельца за каждым merged change, чтобы долг не становился «ничейным».

### Риск 2. Утечка кода через промпты и контекст

**Описание.**
Разработчики могут передавать в внешние модели фрагменты чувствительного кода, внутренних API, конфигураций и бизнес-правил. Утечка происходит не только через прямую отправку файлов, но и через «невинные» куски контекста, по которым можно восстановить архитектуру или логику продукта.

**Вероятность:** средняя.

**Влияние:** критическое.

**Меры снижения.**

1. Политика data classification для промптов, что можно отправлять, что запрещено.
2. Техническое ограничение каналов, корпоративные прокси, allowlist инструментов и блокировка неразрешенных моделей.
3. Автоматическое redaction чувствительных данных перед отправкой контекста.
4. Обучение команды паттернам безопасного prompt engineering.
5. Регулярный аудит промптов в критичных командах и инцидентный процесс для нарушений.

### Риск 3. Логирование prompt/response и вторичная утечка данных

**Описание.**
Даже при корректной работе разработчика риск остается в инфраструктуре логирования. Промпты и ответы могут сохраняться в системах мониторинга, BI, чатах поддержки и аналитике вендора. Это создает вторичный контур хранения чувствительных данных, который часто не учитывается в базовой модели угроз.

**Вероятность:** средняя.

**Влияние:** высокое, в отдельных доменах критическое.

**Меры снижения.**

1. Включить режимы «no training/no retention» там, где это поддерживается.
2. Вести централизованную политику логирования, срок хранения, маскирование, право доступа по роли.
3. Запретить пересылку сырых prompt/response в внешние сервисы без анонимизации.
4. Проводить privacy и security review для всех интеграций AI-платформы.
5. Зафиксировать DLP-контроль и регулярные выборочные проверки журналов.

### Риск 4. Утечка интеллектуальной собственности

**Описание.**
При неуправляемом использовании внешних моделей компания рискует раскрыть элементы своей интеллектуальной собственности, алгоритмы, уникальные эвристики, внутренние модели данных и коммерчески значимые подходы. Даже частичная утечка может снизить конкурентное преимущество.

**Вероятность:** средняя.

**Влияние:** высокое.

**Меры снижения.**

1. Разделить задачи по доменам риска, для чувствительных доменов использовать self-hosted или изолированные модели.
2. Подготовить шаблоны «безопасных промптов» для типовых кейсов.
3. Ввести юридические требования к поставщикам AI-сервисов, включая обработку данных и права на контент.
4. Проводить регулярный контроль рабочих практик через AI Champions и Security.
5. Создать отдельный процесс согласования для задач с высокой IP-ценностью.

### Риск 5. Комплаенс по лицензированию AI-сгенерированного кода

**Описание.**
AI может предлагать фрагменты, которые по структуре или содержанию пересекаются с кодом под лицензиями с ограничениями для коммерческого использования. Если этот риск не контролировать, компания получает претензии по нарушению лицензий и юридические издержки при аудите или сделках.

**Вероятность:** средняя.

**Влияние:** высокое.

**Меры снижения.**

1. Принять формальную политику использования AI-кода и критерии допустимости.
2. Включить license scanning и provenance checks в CI.
3. Требовать от разработчика фиксировать источник и уровень переработки для спорных фрагментов.
4. Организовать регулярные проверки с Legal и Open Source Program Office.
5. Запретить merge для изменений без прохождения лицензирования в high-risk репозиториях.

### Риск 6. Нарушение OSS compliance в цепочке зависимостей

**Описание.**
AI часто предлагает быстрые решения через добавление новых библиотек. Без контроля это ведет к появлению зависимостей с неподходящими лицензиями, уязвимостями или неподдерживаемыми пакетами. Проблема становится особенно дорогой при масштабировании на много команд.

**Вероятность:** высокая.

**Влияние:** высокое.

**Меры снижения.**

1. Централизованный allowlist зависимостей и версий.
2. SCA-проверки и автоматические блокировки в CI/CD.
3. Процесс исключений с обязательным согласованием Security и Legal.
4. SLA на обновление уязвимых пакетов и удаление запрещенных зависимостей.
5. Периодические OSS-аудиты на уровне продукта и платформы.

### Риск 7. Чрезмерная зависимость от AI и деградация инженерных навыков

**Описание.**
Если команда привыкает без проверки принимать ответы модели, снижается глубина понимания системных решений. Особенно уязвимы джуны и инженеры в фазе активного роста. На длинной дистанции это повышает стоимость сопровождения и снижает способность команды решать нестандартные инциденты без подсказки.

**Вероятность:** высокая.

**Влияние:** среднее в краткосроке, высокое в долгосроке.

**Меры снижения.**

1. Встроить в процесс обязательное объяснение решения на ревью, почему оно верно и какие альтернативы отклонены.
2. Сохранить ручные практики для критичных компонентов, design review, отладка, инцидент-анализ.
3. Вести обучение по AI review и архитектурному мышлению, а не только по промптам.
4. В pair-режиме закрепить роли, агент предлагает, инженер принимает решение, reviewer контролирует качество.
5. Отслеживать метрики навыков, доля сложных задач, решенных без прямой генерации.

### Риск 8. Сопротивление изменениям и неравномерное внедрение

**Описание.**
Часть команд может активно использовать AI, часть может игнорировать новый режим. Это приводит к расхождению стандартов, конфликту ожиданий и нестабильному качеству в портфеле продуктов. Организация теряет эффект масштаба.

**Вероятность:** средняя.

**Влияние:** среднее, при росте масштаба высокое.

**Меры снижения.**

1. Запустить единую сеть AI Champions с понятной зоной ответственности.
2. Утвердить минимальный стандарт работы, формат задач, ревью, CI, трассируемость.
3. Проводить пилоты с измеримым результатом и публичным разбором.
4. Встроить adoption-метрики в KPI инженерных менеджеров.
5. Обеспечить поддержку сеньоров в роли архитектурных владельцев, а не пассивных наблюдателей.

### Риск 9. Ошибки в регулируемых зонах, криптография, аутентификация, персональные данные

**Описание.**
В доменах с высоким регуляторным давлением цена ошибки выше обычной. AI может сгенерировать правдоподобный, но небезопасный код в криптографии, ошибочную логику авторизации или некорректную обработку персональных данных. Такие дефекты ведут к инцидентам безопасности, штрафам и репутационному ущербу.

**Вероятность:** средняя.

**Влияние:** критическое.

**Меры снижения.**

1. Выделить regulated zones в отдельный режим разработки с усиленными требованиями.
2. Запретить прямой merge AI-изменений без двойного human review.
3. Сделать обязательными threat modeling, security tests и privacy checks для затронутых компонентов.
4. Использовать только одобренные криптографические примитивы и библиотеки.
5. Применять поэтапный rollout, feature flags и проверенные rollback-процедуры.

## Сводный приоритет управления рисками

Для Year-1 приоритет определяется не только вероятностью, но и потенциальным ущербом. В первую волну снижения должны попасть риски с критическим влиянием, даже если их вероятность не максимальна.

**Приоритет P1, снижать в первую очередь:**

1. Утечка кода через промпты и контекст.
2. Логирование prompt/response и вторичная утечка данных.
3. Ошибки в регулируемых зонах, криптография, аутентификация, персональные данные.
4. Комплаенс по лицензированию AI-сгенерированного кода.

**Приоритет P2, снижать параллельно после запуска P1-контуров:**

1. Усиление технического долга.
2. Нарушение OSS compliance в цепочке зависимостей.
3. Утечка интеллектуальной собственности.

**Приоритет P3, управлять через обучение и организационные механики:**

1. Чрезмерная зависимость от AI и деградация навыков.
2. Сопротивление изменениям и неравномерное внедрение.

## Decision: Which risks to accept, which to mitigate first

**Принимаем осознанно на Year-1:**

- Умеренный рост операционной сложности процесса, больше этапов проверки и формализации.
- Часть временных потерь на обучение и перестройку ролей.
- Ограниченную неравномерность скорости между командами в фазе пилота.

**Не принимаем, снижаем в первую очередь:**

- Любые сценарии утечки кода, данных или IP через AI-контур.
- Любые нарушения в регулируемых зонах, криптография, аутентификация, персональные данные.
- Любые риски лицензирования и OSS compliance, которые могут привести к юридическим последствиям.

**Практическое решение для Year-1:**

1. Сначала строим контур безопасности и комплаенса, политики данных, логирование, лицензии, regulated pipeline.
2. Затем масштабируем AI-режим на большее число команд только после прохождения контрольных метрик.
3. Параллельно стабилизируем качество через архитектурные guardrails и дисциплину ревью, чтобы скорость не превращалась в долг.

Такой порядок дает реалистичный баланс. Команда получает ускорение, но не оплачивает его критичными инцидентами через один или два квартала.

# Self-Hosted инструменты: сравнение и рекомендации

## Резюме для принятия решения

**Рекомендация:** Для Enterprise с ограничением «код не покидает периметр» оптимальный стек — **Tabby** (IDE-ассистент) + **Continue** (гибкий frontend) + **vLLM/Ollama** (локальный inference backend). Альтернативы существуют, но требуют компромиссов.

## Сравнительная таблица self-hosted инструментов

| Инструмент | Категория | Модель развёртывания | Требования к GPU | Лицензия | Зрелость | Ключевые ограничения для strict on-prem |
|---|---|---|---|---|---|---|
| **Tabby** | IDE-ассистент / self-hosted Copilot alternative | Self-hosted server (Docker/K8s), editor plugins | RTX 4090 для пилота, A100/H100 для высокой конкурентности | Apache-2.0 (OSS), отдельная EE лицензия | **Production-ready** (стабильные релизы) | Требует настройки local model ops и indexing |
| **Continue** | IDE-ассистент + CI интеграция | VS Code/JetBrains extension + self-hosted endpoints (Ollama/vLLM) | Зависит от backend (обычно 1x 24GB+ GPU для пилота) | Apache-2.0 | **Production-used**, быстрое развитие | Многие команды используют внешние API без централизованного контроля |
| **Cody (Sourcegraph Enterprise)** | IDE + CLI с enterprise context | Self-hosted Sourcegraph Enterprise + model config | Если Cody Gateway — не требует GPU; self-hosted model — требует GPU | Проприетарная (Sourcegraph EE) | **Enterprise production** | "Fully on-prem" зависит от пути model-provider и настройки policy |
| **Kilo Code** | IDE + CLI агент | Open-source; подключается к local endpoints | Backend-зависим | MIT | **Active, high velocity** | Часто API-first; strict perimeter требует policy enforcement |
| **OpenCode** | CLI агент | Local terminal agent; поддерживает LOCAL_ENDPOINT | Backend-зависим | MIT | **Early / transitioned** (оригинальный repo archived) | Риск долгосрочной поддержки; migration path неясен |
| **Claude Code** | CLI/IDE агент | Anthropic surfaces + enterprise controls | Не требует local GPU (provider-hosted) | Проприетарная | **Enterprise-ready** | **Не fully self-hosted inference**; требует enterprise controls для compliance |
| **PR-Agent** | Code review бот | Self-hosted via GitHub Action/CLI/Docker/webhook | Низкие (inference backend определяет GPU) | AGPL-3.0 | Широко используется | AGPL лицензия; качество зависит от LLM backend |
| **Qodo Cover** | Test generation | CLI/CI интеграция | Низкие (model backend определяет GPU) | AGPL-3.0 | **Maintenance risk** (repo не поддерживается) | Статус поддержки и AGPL constraints |

## Local LLM бэкенды (для perimeter-safe deployment)

| Бэкенд | Лучше всего подходит | Инфраструктура | Enterprise замечания |
|---|---|---|---|
| **Ollama** | Быстрый локальный rollout, пилоты | CPU или single-GPU; Docker images с CPU/NVIDIA/AMD | Сильная сторона — скорость пилота; меньше контроля на hyperscale чем vLLM |
| **vLLM** | High-throughput shared inference | GPU-centric serving; сильная batching/throughput | Лучше для centralized enterprise inference tier; выше SRE/K8s complexity |
| **llama.cpp** | Edge/on-device/air-gapped | Может работать CPU-only; поддерживает CUDA/Metal/Vulkan | Отлично для hard perimeter и low-footprint deployments; ниже throughput для multi-user |

### Практические GPU рекомендации

- **Пилот (10–30 разработчиков):** 1–2 inference nodes, RTX 4090 / L40S
- **Департамент (50–150 разработчиков):** Shared inference service, A100 80GB / H100
- **Enterprise platform (100+ concurrent users):** Multi-node vLLM с autoscaling, H100/A100 pools

---

## Категорийные рекомендации

### A. IDE-ассистенты

**Primary self-hosted candidates:**
- **Tabby** — наиболее прямой on-prem Copilot alternative
- **Continue** — наиболее гибкий frontend, backend-agnostic
- **Cody Enterprise** — если Sourcegraph уже strategic platform

### B. CLI-агенты

**Primary perimeter-safe pattern:**
- **Kilo/OpenCode-style CLI** или equivalent agent frontends, подключённые к **local OpenAI-compatible endpoints** (vLLM/Ollama gateway)

**Conditional/non-primary:**
- **Codex CLI / Claude Code** — только если governance принимает managed-provider path с approved gateway controls

### C. Code review боты

- **PR-Agent** для self-hosted PR analysis в CI
- Использовать local model gateway для сохранения diffs/code в периметре

### D. Test generation

- **Qodo Cover** концептуально подходит, но maintenance status требует caution
- Предпочесть actively maintained alternatives для long-term support

## Evaluation rubric (weighted, enterprise-focused)

| Критерий | Вес | Что значит "хорошо" |
|---|---|---:|:---|
| **Data residency / perimeter compliance** | **25%** | Full on-prem operation, auditable no-exfil path, policy enforcement |
| **Integration with existing SDLC** | **15%** | IDE/CI/VCS integration с низкой стоимостью change-management |
| **Model backend flexibility** | **12%** | Native support для Ollama/vLLM/OpenAI-compatible endpoints |
| **Operational maturity** | **12%** | Stable releases, active maintainers, clear upgrade path |
| **Security/governance controls** | **10%** | RBAC, logs, policy, approvals, admin controls |
| **Developer experience / adoption speed** | **10%** | Good UX, low friction onboarding, low prompt overhead |
| **Performance at team scale** | **8%** | Acceptable latency при realistic concurrency |
| **License and legal fit** | **8%** | License совместимая с enterprise policy (включая AGPL review) |

### Pass/fail gates перед weighted scoring

1. **Perimeter compliance gate:** Инструмент должен поддерживать полностью локальную работу без обязательных внешних вызовов.
2. **Active maintenance gate:** Публичный репозиторий с активностью в последние 3 месяца или enterprise vendor support contract.
3. **License compatibility gate:** License проходит legal review (особое внимание к AGPL в enterprise контексте).

## Рекомендуемые стеки (варианты)

### Минимальный стек (пилот, быстрый старт)
- **IDE:** Continue + Ollama на рабочих станциях разработчиков
- **Backend:** Ollama с Llama 3.1/CodeLlama
- **GPU:** RTX 4090 или CPU-only для пилота
- **Cost:** $3-5k на GPU, 0 на лицензии
- **Time to value:** 1-2 дня

### Оптимальный стек (департамент, продакшн)
- **IDE:** Tabby (server) + Continue (frontend)
- **Backend:** vLLM на shared inference cluster
- **GPU:** A100 80GB x 2-4 для команды 50-100 человек
- **Code review:** PR-Agent + local model gateway
- **Cost:** $50-100k на GPU + $10-20k на инфраструктуру
- **Time to value:** 2-4 недели

### Максимальный стек (enterprise platform)
- **IDE:** Tabby Enterprise + Cody (если Sourcegraph уже используется)
- **Backend:** Multi-node vLLM с autoscaling, model routing
- **GPU:** H100/A100 pools, 8+ GPUs
- **Code review:** PR-Agent с централизованным gateway
- **Test generation:** Self-hosted alternatives к Qodo Cover
- **Cost:** $200-500k на GPU + $50-100k на инфраструктуру + лицензии
- **Time to value:** 2-3 месяца

---

## Решение: Какой стек выбрать и сколько это стоит

### Для начала (Year 1)

**Рекомендация:** Начать с **оптимального стека** — он даёт правильный баланс между time-to-value и enterprise readiness.

**Последовательность:**
1. Пилот с Continue + Ollama (1-2 недели, proof of concept)
2. Переход на Tabby + vLLM для production (2-4 недели)
3. Добавление PR-Agent для code review (1-2 недели)
4. Масштабирование на enterprise platform по мере роста команды

**Budget Year 1:**
- GPU infrastructure: $50-150k (зависит от размера команды)
- Software licenses: $0-30k (Tabby OSS бесплатен, Sourcegraph/Cody платный)
- SRE/DevOps effort: 0.5-1 FTE на 3-6 месяцев
- Training and adoption: включено в общий training budget

### Trade-offs

| Подход | Pros | Cons |
|---|---|---|
| **Минимальный** | Быстрый старт, низкая стоимость | Не масштабируется, performance issues при росте команды |
| **Оптимальный** | Правильный баланс, production-ready | Требует инвестиций в GPU и SRE |
| **Максимальный** | Enterprise-grade, полный контроль | Высокая стоимость, длительный time-to-value |

**Вывод:** Для Enterprise с 100+ разработчиками оптимальный стек — лучший выбор для Year 1. Минимальный подходит только для proof of concept, максимальный — для mature organizations с существующей ML Ops инфраструктурой.

## Приложение B: AI Readiness Checklist (расширенный)

### Инженерные практики

- [ ] **One obvious way** — Для каждой типовой задачи есть один предпочтительный путь реализации
- [ ] **Тесты защищают критические flows** — Важные сценарии пользователя покрыты тестами
- [ ] **Build/lint/test работают без догадок** — Команды явно задокументированы и воспроизводимы
- [ ] **Deprecated patterns помечены** — Старые подходы явно отмечены как не рекомендуемые
- [ ] **Дубли и конфликтующие абстракции сокращены** — Нет нескольких "правильных" реализаций одной задачи
- [ ] **Документация описывает архитектуру** — Есть связь между кодом, решением и бизнес-смыслом
- [ ] **CI ловит типовые регрессии** — Основные проверки качества автоматизированы и обязательны

### AI-specific практики

- [ ] **AGENTS.md создан** — В корне репозитория есть файл с HOW/WHY/WHAT
- [ ] **Context packing документирован** — Понятно, какие файлы нужны для типовых задач
- [ ] **ADR для ключевых решений** — Архитектурные решения документированы с причинами
- [ ] **Deprecated zones помечены** — Код, который не должен копироваться AI, явно помечен
- [ ] **Guardrails настроены** — Линтинг, type checking, тесты блокируют merge при нарушениях
- [ ] **Code review для AI-generated кода** — Есть чеклист для review AI-generated изменений

### Организационная готовность

- [ ] **Baseline метрики собраны** — Есть данные до внедрения AI для сравнения
- [ ] **Пилотная команда выбрана** — Есть команда для первого внедрения
- [ ] **Champions network** — Выделены энтузиасты для помощи другим командам
- [ ] **Training program** — План обучения команд работе с AI
- [ ] **Governance policy** — Понятно, кто отвечает за AI-generated код

### Self-assessment по уровням зрелости

**Уровень 1 — Chaotic (0-2 пункта)**
- Риск: AI ускоряет накопление техдолга

**Уровень 2 — Basic (3-4 пункта)**
- Риск: Локальные улучшения есть, но масштабирование ломается

**Уровень 3 — Structured (5 пунктов)**
- Эффект: Заметный рост скорости без резкого роста дефектов

**Уровень 4 — Optimized (6 пунктов)**
- Эффект: AI-ассистенты работают предсказуемо, доля rework снижается

**Уровень 5 — AI-Native (7+ пунктов)**
- Эффект: Высокая скорость поставки и управляемое качество на уровне организации

│       ├── api/

│   ├── components/                # UI kit
│   ├── hooks/                     # Общие hooks
│   ├── lib/                       # Утилиты
│   └── types/                     # Shared types
│

    ├── layout.tsx
    ├── providers.tsx
    └── routes.ts

### Почему это AI-friendly

1. **Все файлы фичи в одном месте** — AI видит полный контекст без прыжков по директориям
2. **Предсказуемая структура** — Каждая фича следует одному шаблону
3. **README в каждой фиче** — Документация рядом с кодом
4. **Четкие границы** — Минимум shared кода, максимум явных зависимостей

### Input Metrics (Adoption)

| Метрика | Источник данных | Частота сбора | Целевое значение |

| Adoption rate | Tool analytics / surveys | Еженедельно | > 80% команды использует |
| Usage frequency | IDE plugin logs / CLI logs | Ежедневно | > 5 сессий на разработчика в день |
| % AI-assisted PRs | Git provider API (labels/comments) | Еженедельно | > 30% PRs с AI-помощью |
| Lines of code generated | IDE/CLI analytics | Еженедельно | Тренд, не цель |

### Output Metrics (Productivity)

| Метрика | Источник данных | Частота сбора | Целевое значение |
|---|---|---|---|
| PR velocity (time to merge) | Git provider API | Еженедельно | -20% vs baseline |
| Cycle time | Git provider API / issue tracker | Еженедельно | -15% vs baseline |
| Test coverage | CI reports | На каждый PR | Не снижать ниже baseline |
| Rework rate (code churn) | Git provider API | Еженедельно | < 15% изменений в первую неделю |
| Build success rate | CI logs | Ежедневно | > 95% |

| Метрика | Источник данных | Частота сбора | Целевое значение |

| Defect rate (production bugs) | Bug tracker | Еженедельно | Не выше baseline |
| Developer satisfaction | Surveys (eNPS) | Ежеквартально | > 7/10 |
