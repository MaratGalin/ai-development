# Запрос входных данных от команды разработки для старта multi-agent проекта

## Зачем нужен этот документ

Этот документ нужен, чтобы заранее собрать от команды разработки ПО минимальный и рекомендованный набор входных данных для запуска проекта, который будет разрабатываться в **мультиагентном режиме** с использованием **Oh My OpenAgent**.

Цель простая: снизить количество догадок, сократить число лишних итераций и дать агентам и людям общий, однозначный контекст для старта работы.

Важно: мы не просим команду сразу готовить «полное ТЗ на всё». Для старта достаточно **минимального пакета**, а детальные артефакты можно дослать по мере проработки.

---

## Что обязательно нужно получить на старте

### 1. Цели проекта и границы scope

Нужно запросить:

- бизнес-цель проекта;
- целевых пользователей;
- ожидаемую ценность результата;
- критерии успеха;
- что входит в первый релиз / первый инкремент;
- что **явно не входит** в scope на старте.

Почему это важно:

- агенты и команда не будут расширять scope по догадке;
- проще планировать вертикальные срезы;
- проще принимать решения по приоритетам.

### 2. Функциональные требования

Нужно запросить:

- приоритизированный список фич / use cases;
- acceptance criteria по ключевым фичам;
- владельца приемки результата по каждой крупной области.

Почему это важно:

- мультиагентная разработка лучше работает на конкретных, проверяемых задачах;
- это позволяет связывать фичу, критерии готовности и тесты.

### 3. Архитектурные ожидания и ограничения

Нужно запросить:

- какую архитектуру команда привыкла видеть или считает предпочтительной;
- какие паттерны обязательны;
- какие технологии нежелательны или запрещены;
- какие архитектурные решения уже приняты и не должны пересматриваться;
- если есть — ADR, архитектурные заметки, схемы компонентов.

Почему это важно:

- агенты не будут предлагать архитектурно «красивые», но контекстно неверные решения;
- снижается риск споров по базовой структуре проекта.

### 4. Доменная модель

Нужно запросить:

- основные сущности;
- связи между сущностями;
- доменные термины и глоссарий;
- критичные инварианты и ограничения;
- если есть — **доменные графы**, bounded contexts, state diagrams.

Почему это важно:

- доменная неоднозначность — один из главных источников ошибок у AI-агентов;
- инварианты помогают не ломать бизнес-логику при реализации.

### 5. Бизнес-флоу и process flows

Нужно запросить:

- 3–5 ключевых end-to-end сценариев;
- happy path;
- exception flows / edge cases;
- ошибки, откаты, retry-сценарии, ручную обработку;
- если есть — BPMN, sequence diagrams, user flows, state diagrams.

Почему это важно:

- без exception flows и пограничных сценариев система почти всегда проектируется слишком оптимистично;
- мультиагенты лучше работают, когда путь процесса явно описан.

### 6. Интеграции и внешние зависимости

Нужно запросить:

- список внешних систем;
- API / контракты / схемы событий;
- форматы обмена;
- auth / permissions / rate limits / sandbox;
- владельца каждой интеграции.

Почему это важно:

- интеграции часто становятся главным источником скрытых ограничений;
- без контрактов агенты начинают «додумывать» интерфейсы.

### 7. Нефункциональные требования

Нужно запросить:

- требования по безопасности;
- производительности;
- SLA / SLO;
- аудиту и логированию;
- compliance / регуляторным ограничениям;
- хранению, удалению и маскированию данных.

Почему это важно:

- такие ограничения редко угадываются из функционального описания;
- именно они часто определяют архитектуру сильнее, чем функциональные сценарии.

### 8. Критерии качества и готовности

Нужно запросить:

- какие проверки обязательны перед merge / release;
- какие quality gates обязательны;
- какие команды / проверки считаются минимально достаточными;
- Definition of Ready;
- Definition of Done;
- требования к code review.

Почему это важно:

- агенты должны понимать не только что делать, но и как доказать, что результат готов;
- это снижает rework и спорные трактовки «готовности».

### 9. Ownership и эскалация решений

Нужно запросить:

- кто принимает продуктовые решения;
- кто принимает архитектурные решения;
- кто отвечает за интеграции;
- кто подтверждает приемку;
- ожидаемый SLA ответа на блокеры и открытые вопросы.

Почему это важно:

- в multi-agent разработке узкое место часто не код, а зависшие уточнения;
- явные decision owners позволяют быстро снимать неоднозначности.

---

## Что желательно получить в течение первой недели

Это не обязательно для самого первого старта, но сильно снижает риск лишних итераций:

- system context diagram;
- схема ключевых компонентов;
- repository map или ожидаемая структура проекта;
- схема ролей и прав доступа;
- ERD / data model diagram;
- OpenAPI / AsyncAPI / event contracts;
- ADR по ключевым решениям;
- примеры похожих систем или эталонных реализаций;
- список известных рисков, assumptions и TBD-вопросов;
- перечень source-of-truth документов.

---

## Особенно полезные артефакты

Если у команды это уже есть, стоит запросить в первую очередь:

1. **Product brief / PRD-lite** по ключевым сценариям.
2. **Приоритизированный backlog** с acceptance criteria.
3. **ADR** по архитектурным развилкам.
4. **Domain glossary**.
5. **Domain graph / bounded context map**.
6. **BPMN / sequence diagrams / user flows**.
7. **OpenAPI / AsyncAPI / event schemas**.
8. **Quality gates / Definition of Done**.
9. **Ownership matrix**.
10. **Source-of-truth map** — если документы расходятся, чему верим в первую очередь.

---

## Что часто забывают запросить, но это критично

### Non-goals / out of scope

Нужно явно зафиксировать, что **не входит** в первую фазу. Иначе и люди, и агенты начинают достраивать систему шире, чем требуется.

### Source of truth

Если есть несколько документов, нужно заранее понимать, какой из них главный:

- PRD,
- backlog,
- Confluence,
- Swagger,
- Miro,
- ADR,
- Jira.

### Exception flows

Нужно просить не только happy path, но и:

- таймауты;
- отказ внешней системы;
- конфликты состояния;
- повторные попытки;
- ручную обработку;
- rollback-сценарии.

### Инварианты домена

Примеры:

- заказ не может быть оплачен дважды;
- пользователь не может иметь две активные подписки одного типа;
- статус X допустим только после статуса Y.

Именно такие ограничения особенно полезны для AI-агентов.

### Unknowns list

Если часть информации ещё не определена, достаточно запросить:

- пометку **TBD**;
- владельца;
- ожидаемую дату появления решения.

Это лучше, чем неявные пробелы в требованиях.

## Готовое письмо команде разработки

**Тема:** Что нужно подготовить для старта multi-agent разработки проекта

Коллеги,

чтобы мы смогли быстро и без лишних итераций запустить разработку проекта в мультиагентном режиме на базе **Oh My OpenAgent**, просим прислать **минимальный пакет входных данных по проекту**.

Можно в виде ссылок на уже существующие документы, схемы, backlog, Confluence, Miro, Swagger, ADR и т.д. — заново переписывать ничего не нужно.

### 1. Что обязательно нужно на старте

**1) Цели и границы проекта**
- бизнес-цель проекта;
- для кого делаем продукт / кто основные пользователи;
- критерии успеха;
- что входит в первый релиз / инкремент;
- что явно не входит в scope на старте.

**2) Функциональные требования**
- приоритизированный список фич / use cases;
- acceptance criteria по каждой ключевой фиче;
- кто принимает результат со стороны бизнеса / продукта.

**3) Архитектурные ожидания и ограничения**
- какую архитектуру команда считает предпочтительной;
- есть ли обязательные паттерны, которые нужно сохранить;
- есть ли запреты по стеку / технологиям;
- какие архитектурные решения уже приняты и не должны пересматриваться;
- если есть ADR или аналогичные записи — просим приложить.

**4) Доменная модель**
- ключевые сущности и их связи;
- доменные термины / глоссарий;
- критичные инварианты и бизнес-правила;
- если есть — доменные графы / схемы предметной области.

**5) Основные бизнес-флоу / process flows**
- 3–5 ключевых end-to-end сценариев;
- happy path;
- exception flows / edge cases / ошибки / откаты;
- если есть BPMN, sequence diagram, user flow, state diagram — это особенно полезно.

**6) Интеграции и внешние зависимости**
- какие внешние системы участвуют;
- API / контракты / форматы обмена;
- auth / permissions / rate limits / sandbox;
- кто владелец интеграции со стороны команды.

**7) Нефункциональные требования**
- безопасность;
- производительность;
- SLA / SLO;
- аудит / логирование;
- compliance / регуляторные ограничения;
- требования к хранению и удалению данных.

**8) Качество и критерии готовности**
- какие проверки обязательны перед merge / release;
- test/lint/typecheck/build/security checks;
- Definition of Done / что считается завершённой задачей;
- есть ли обязательные правила code review.

**9) Ownership и правила эскалации**
- кто принимает архитектурные решения;
- кто отвечает за продуктовые уточнения;
- кто отвечает за интеграции;
- кто отвечает за приемку;
- ожидаемый SLA ответа на блокеры.

### 2. Что желательно прислать в первую неделю

- карта системы / system context diagram;
- схема ключевых компонентов;
- repository map / структура будущего проекта, если уже есть ожидания;
- пример похожей системы, на которую хотим ориентироваться;
- примеры API-контрактов;
- схема ролей и прав доступа;
- схема данных / ERD;
- шаблоны документации, если в команде есть preferred format;
- список известных рисков, допущений и TBD-вопросов.

### 3. Особенно полезные артефакты

Если это уже существует, будет очень полезно получить:

- PRD / Feature brief по ключевым сценариям;
- ADR по важным архитектурным решениям;
- Domain glossary;
- Domain graph / bounded contexts / state diagrams;
- BPMN / sequence diagrams / user flows;
- OpenAPI / AsyncAPI / event contracts;
- Definition of Ready / Definition of Done;
- список quality gates;
- ссылки на исходные документы, которые считать source of truth.

### 4. Если чего-то пока нет

Если по части пунктов артефакты ещё не готовы, достаточно:

- пометить как **TBD**;
- указать **владельца**;
- указать **ориентировочную дату**, когда информация будет готова.

Так мы сможем стартовать работу по вертикальным срезам без ожидания полного пакета документации и параллельно закрывать пробелы.

Спасибо.

## Короткая версия письма

Если нужен совсем компактный вариант, можно использовать следующий текст:

Коллеги, для старта multi-agent разработки проекта просим прислать:

1. цели проекта, scope и non-goals;
2. приоритетные фичи с acceptance criteria;
3. архитектурные ожидания, ограничения и принятые решения;
4. доменную модель: сущности, термины, инварианты;
5. ключевые process flows: happy path + exception flows;
6. интеграции и контракты;
7. нефункциональные требования;
8. quality gates и критерии готовности;
9. владельцев решений и SLA по блокерам.

Если есть — отдельно очень полезны: ADR, domain graph, BPMN/sequence diagrams, схема данных, роли/права, OpenAPI/AsyncAPI, ссылки на source-of-truth документы.

---

## Итоговая рекомендация

Лучший формат запроса к команде — не «пришлите всё, что знаете о проекте», а:

- **обязательный минимум для старта**;
- **рекомендуемые артефакты на первую неделю**;
- **явное указание владельцев неизвестных вопросов**.

Именно такой формат лучше всего подходит для мультиагентной разработки: он даёт достаточно контекста для старта, но не превращает подготовку в тяжёлый бюрократический этап.
