# Формальный запрос входных материалов для старта multi-agent разработки проекта

## Назначение документа

Данный документ предназначен для подготовки официального запроса в адрес команды разработки программного обеспечения с целью сбора исходных материалов, необходимых для запуска проекта в **мультиагентном режиме разработки** с использованием **Oh My OpenAgent**.

Документ решает две задачи:

1. Формализует перечень материалов, которые необходимы для корректного старта проектирования и разработки.
2. Дает расшифровку терминов, сокращений и типов артефактов, чтобы получатели письма одинаково понимали, что именно требуется предоставить.

Ключевой принцип: на старте необходим **не полный массив документации**, а достаточный, структурированный набор материалов, позволяющий избежать неоднозначности в требованиях, архитектуре, доменной логике и критериях приемки.

---

## Рекомендуемый официальный текст письма

**Тема:** Запрос исходных материалов для запуска проекта в multi-agent режиме разработки

Уважаемые коллеги,

В рамках подготовки к запуску нового проекта, который планируется реализовывать в **multi-agent режиме разработки** с применением платформы **Oh My OpenAgent**, просим предоставить исходные материалы, необходимые для корректного старта архитектурного и инженерного контура работ.

Цель данного запроса — сформировать единое и однозначное понимание:

- целей проекта;
- границ функционального объема;
- архитектурных ожиданий;
- доменной модели;
- ключевых бизнес-процессов;
- интеграционных ограничений;
- требований к качеству и приемке результата.

Просим по возможности направить материалы в виде ссылок на уже существующие документы, схемы, backlog, корпоративную wiki, Confluence, Jira, Swagger/OpenAPI, ADR и иные артефакты. Подготовка новых документов с нуля не требуется, если релевантная информация уже существует в проектном контуре.

### 1. Материалы, необходимые для старта работ

#### 1.1. Цели проекта и границы реализации

Просим предоставить:

- бизнес-цели проекта;
- описание целевых пользователей или групп пользователей;
- ожидаемый бизнес-результат;
- критерии успешности проекта или первого этапа;
- функциональные границы первого релиза / инкремента;
- перечень того, что **не входит** в рамки стартового этапа.

#### 1.2. Функциональные требования

Просим предоставить:

- приоритизированный перечень функций, сценариев или use cases;
- критерии приемки по ключевым функциям;
- указание ответственных лиц, подтверждающих корректность реализации со стороны бизнеса или продукта.

#### 1.3. Архитектурные ожидания и ограничения

Просим предоставить:

- предпочтительный архитектурный подход;
- обязательные архитектурные или технологические паттерны;
- ограничения по стеку технологий;
- перечень архитектурных решений, которые уже приняты и не подлежат пересмотру в рамках текущего этапа;
- при наличии — архитектурные схемы, описания компонентов, записи архитектурных решений.

#### 1.4. Доменная модель

Просим предоставить:

- основные бизнес-сущности;
- связи между сущностями;
- глоссарий ключевых терминов;
- критичные бизнес-правила и инварианты;
- при наличии — доменные схемы, domain graph, bounded context map, state model.

#### 1.5. Ключевые бизнес-процессы и сценарии

Просим предоставить:

- 3–5 ключевых end-to-end сценариев;
- основные пользовательские или системные потоки;
- сценарии обработки ошибок и исключительных ситуаций;
- сценарии повторных попыток, откатов, ручной обработки;
- при наличии — BPMN, sequence diagrams, user flows, state diagrams.

#### 1.6. Интеграции и внешние зависимости

Просим предоставить:

- перечень внешних систем и сервисов;
- API-контракты, event contracts, форматы обмена данными;
- требования к авторизации и разграничению доступа;
- ограничения по частоте вызовов, sandbox-окружения, тестовые стенды;
- владельцев интеграций со стороны команды.

#### 1.7. Нефункциональные требования

Просим предоставить:

- требования по безопасности;
- требования по производительности;
- SLA и/или SLO, если применимо;
- требования к аудиту, логированию, наблюдаемости;
- регуляторные и compliance-ограничения;
- требования к хранению, удалению, маскированию и защите данных.

#### 1.8. Требования к качеству и готовности результата

Просим предоставить:

- перечень обязательных quality gates;
- требования к тестированию и верификации;
- Definition of Ready и Definition of Done, если приняты в команде;
- обязательные проверки перед merge / release;
- правила code review и приемки результата.

#### 1.9. Ответственные лица и порядок эскалации

Просим предоставить:

- ответственных за продуктовые решения;
- ответственных за архитектурные решения;
- ответственных за интеграции;
- ответственных за приемку результата;
- ожидаемый срок ответа на уточняющие вопросы и блокеры.

### 2. Материалы, рекомендуемые к предоставлению в первую неделю

Дополнительно рекомендуем направить:

- system context diagram;
- схему ключевых компонентов;
- repository map или ожидаемую структуру проекта;
- ERD / data model diagram;
- схемы ролей и прав доступа;
- OpenAPI / AsyncAPI спецификации;
- ADR по ключевым решениям;
- перечень известных рисков, assumptions и TBD-вопросов;
- список документов, которые следует считать source of truth.

### 3. Если часть материалов отсутствует

Если отдельные материалы на текущий момент отсутствуют, просим по возможности указать:

- что именно находится в статусе **TBD**;
- кто является владельцем данного вопроса;
- ориентировочный срок предоставления информации.

Такой формат позволит начать работу по приоритетным вертикальным срезам без ожидания полной финализации всех проектных артефактов.

С уважением,

[Имя / подразделение / роль]

## Пояснения к сокращениям, терминам и типам артефактов

Ниже приведены пояснения, которые можно приложить к письму или использовать во внутренней коммуникации, чтобы получатели одинаково понимали запрашиваемые материалы.

### Multi-agent разработка

Под multi-agent разработкой понимается подход, при котором в инженерном процессе участвует не один универсальный AI-ассистент, а несколько специализированных агентных ролей: поиск контекста, работа с документацией, анализ, планирование, проверка и исполнение.

### Oh My OpenAgent

Oh My OpenAgent — это open-source фреймворк для мультиагентной инженерной работы, в котором разные агентные роли используются согласованно и по формальным правилам: поиск, анализ, реализация, проверка, handoff.

### Use case

Use case — это пользовательский или системный сценарий, описывающий, какую задачу нужно решить и какой результат должен быть получен.

### End-to-end сценарий

End-to-end сценарий — это полный поток выполнения процесса от входной точки до финального результата, включая основные переходы, проверки и реакцию системы.

### Acceptance criteria

Acceptance criteria — это критерии приемки. Они определяют, по каким признакам можно однозначно подтвердить, что конкретная функция реализована корректно и соответствует ожиданиям.

### Scope

Scope — это границы задачи или проекта: что входит в текущую фазу работ, а что в нее не входит.

### Non-goals / out of scope

Non-goals или out of scope — это перечень того, что **осознанно не реализуется** на текущем этапе. Это помогает избежать расползания требований и неверных ожиданий.

### Domain model

Domain model — это модель предметной области: сущности, их состояния, связи между ними, а также правила, которые определяют допустимое поведение системы.

### Domain glossary

Domain glossary — это глоссарий доменных терминов. Он нужен для того, чтобы одинаковые понятия во всех документах и обсуждениях трактовались одинаково.

### Domain graph

Domain graph — это графическое представление доменной модели: сущностей, связей, зависимостей и переходов между состояниями. Это может быть схема в Miro, UML-диаграмма, bounded context map или любая другая визуальная модель.

### Bounded context map

Bounded context map — это схема границ отдельных доменных контекстов в системе: где заканчивается одна область ответственности и начинается другая, какие между ними интерфейсы и зависимости.

### Invariants

Invariants — это инварианты, то есть правила, которые не должны нарушаться системой ни при каких обстоятельствах. Например: заказ не может быть оплачен дважды.

### BPMN

**BPMN (Business Process Model and Notation)** — это стандарт визуального описания бизнес-процессов. Обычно используется для отображения шагов процесса, ролей, развилок, исключений и условий переходов.

Если у команды нет BPMN-схем, подойдут более простые блок-схемы, user flows или sequence diagrams.

### Sequence diagram

Sequence diagram — это диаграмма последовательностей, которая показывает, какие участники процесса взаимодействуют друг с другом и в каком порядке.

### User flow

User flow — это схема пользовательского сценария: как пользователь проходит через интерфейс или процесс от действия к результату.

### State diagram / state model

State diagram — это схема состояний объекта или процесса. Она показывает, в каких состояниях может находиться сущность и какие события переводят ее из одного состояния в другое.

### API contract

API contract — это формализованное описание интерфейса взаимодействия между системами: методы, параметры, структуры запросов и ответов, ошибки, ограничения.

### Event contract

Event contract — это описание событийного взаимодействия: какие события публикуются, в каком формате, кем, при каких условиях и как они должны интерпретироваться потребителями.

### OpenAPI

**OpenAPI** — это стандарт машинно- и человеко-читаемого описания HTTP API. Часто используется для документирования REST API. В некоторых командах такие материалы также называют Swagger-спецификациями.

Если отдельной OpenAPI-спецификации нет, подойдут ссылки на Swagger UI, Postman collections или актуальные описания API в wiki.

### AsyncAPI

**AsyncAPI** — это стандарт описания событийных, асинхронных и message-driven интерфейсов, например при работе через брокеры сообщений или event bus.

### ADR

**ADR (Architecture Decision Record)** — это краткая запись архитектурного решения. Обычно включает:

- контекст проблемы;
- рассматриваемые варианты;
- выбранное решение;
- последствия и компромиссы.

Если формальных ADR нет, подойдут архитектурные заметки, RFC, decision logs или иные документы, где зафиксировано, почему было принято то или иное решение.

### Repository map

Repository map — это краткая карта структуры репозитория: где расположены основные модули, документация, тесты, интеграции, конфигурации и исходные точки входа.

### Source of truth

Source of truth — это приоритетный источник истины. Если разные документы противоречат друг другу, важно заранее понимать, какому из них следует доверять в первую очередь.

### ERD

**ERD (Entity-Relationship Diagram)** — это диаграмма сущностей и связей. Она нужна для понимания логической структуры данных.

### Data model diagram

Data model diagram — это схема данных. Она может быть как логической, так и физической: сущности, таблицы, атрибуты, связи, ограничения.

### SLA

**SLA (Service Level Agreement)** — это согласованный уровень сервиса, например требования по времени реакции или доступности.

### SLO

**SLO (Service Level Objective)** — это целевое значение качества сервиса, например 99.9% доступности или определенный предел времени отклика.

### Quality gates

Quality gates — это обязательные контрольные проверки, которые должны быть пройдены до приемки изменений. Например:

- прохождение тестов;
- отсутствие критичных замечаний линтера;
- успешная сборка;
- успешный typecheck;
- прохождение security checks.

### Definition of Ready

Definition of Ready — это набор критериев, по которым задача считается достаточно подготовленной к началу разработки.

Например:

- понятна цель;
- известны ограничения;
- есть критерии приемки;
- определены ответственные лица.

### Definition of Done

Definition of Done — это набор критериев, по которым задача считается завершенной.

Например:

- функциональность реализована;
- тесты пройдены;
- документация обновлена;
- изменения приняты ответственными лицами.

### Code review

Code review — это процесс проверки изменений другими инженерами или ответственными лицами до их включения в основную ветку разработки.

### Assumptions

Assumptions — это допущения, которые принимаются как временно истинные, пока не получено окончательное подтверждение.

### TBD

**TBD (To Be Determined)** — это пометка о том, что вопрос еще не закрыт и решение будет определено позднее.

## Практическое пояснение: что можно прислать вместо «идеального артефакта»

Если в команде нет формализованных документов в нужном формате, можно предоставить упрощенные аналоги:

- вместо ADR — письмо, RFC, страница в wiki или заметка с описанием принятого решения и его причин;
- вместо BPMN — блок-схема, Miro-диаграмма или просто визуально оформленный user flow;
- вместо OpenAPI — Swagger UI, Postman collection или актуальный API reference;
- вместо ERD — схема таблиц, draw.io-диаграмма или описание связей в wiki;
- вместо formal glossary — страница с терминами и их определениями;
- вместо repository map — краткое описание структуры будущего или текущего решения.

Главное требование — не формальный вид артефакта, а его способность снижать неоднозначность и давать единый контекст для проектирования и разработки.

## Итоговая рекомендация

Для enterprise-коммуникации рекомендуется использовать следующую рамку:

1. **Минимально необходимый пакет для старта** — то, без чего нельзя безопасно начать проектирование и реализацию.
2. **Рекомендуемый пакет в течение первой недели** — то, что существенно снижает риски rework.
3. **Явная фиксация неизвестных вопросов** — что отсутствует, кто владелец, когда ожидается решение.

Именно такой подход обеспечивает баланс между управляемостью, скоростью старта и достаточной инженерной определенностью для multi-agent разработки.
